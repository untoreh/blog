Emacs

Por qué emacs y mi lista de deseos personal de emacs.

¿Qué es emacs?

A máquina virtual o intérprete para el emacs lenguaje de programación lisp ... algo así. El lenguaje de programación emacs lisp (elisp) es de propósito general, pero tiene soporte de primera clase para el editor de texto real que lo ejecuta. Los tipos primitivos se centran en la edición de texto, sin embargo, puede escribir lo que quiera con bonitos rendimiento decente (recientemente gracias a la compilación jit ) porque es un descendiente ceceo de MACLISP y un hermano de ceceo común . Sin embargo emacs lisp no tiene un estándar, y la especificación es equivalente a la implementación más popular, GNU Emacs.

¿Por qué emacs?

Con la frase "Emacs es un sistema operativo", la gente quiere decir que puede usarlo para hacer cualquier cosa para la que usaría una computadora. No llega a ser un sistema operativo literal ya que el kernel (generalmente Linux) todavía está a cargo del hardware.[1] Sin embargo, se puede llamar fácilmente un entorno de escritorio, ya que proporciona un entorno con el que trabajar. Le brinda la capacidad de escribir código interactivo rápido para interactuar con cualquier aplicación, puede ser tan sucio como los scripts de shell o bien pensado para proporcionar API estables. De hecho, los paquetes proporcionan muchas funcionalidades de emacs.[2] ¿Pueden otros editores ser tan ampliables como emacs? ¿No porque? Debido a que otros editores utilizan un modelo de usuario diferente, uno en el que el usuario tiene que estar restringido, ya que se considera un "invitado" del entorno de ejecución que ofrece un "servicio" al usuario. Este no es el caso de emacs, donde el entorno y el usuario se vuelven uno y el mismo. En emacs, tiene acceso a todo y puede modificar casi todas las cosas a través de los mismos medios que se crearon, también conocido como elisp.

¿Vale la pena?

Los beneficios se acumulan, es una inversión y, como cualquier inversión, debe esperar ganancias proporcionales a la cantidad que invierte, por lo tanto, debe dedicar algo de tiempo a aprender y practicar cómo usarlo.

Mi lista de deseos de GNU emacs

Emacs es un muy buen sistema operativo, pero un mal editor de texto.

...O algo por el estilo. La implementación de GNU Emacs, al igual que emacs, tiene sus raíces en los años ochenta, y muchas partes de su núcleo muestran su antigüedad ...

  • La dicotomía entre la interfaz gráfica y la interfaz del terminal. Dado que emacs es antiguo, comenzó como una aplicación para ejecutarse dentro de una terminal, pero con la llegada de las interfaces gráficas en los años noventa, comenzó a recopilar funcionalidades más orientadas a basado en ventana interacciones y alrededor del año 2000 el apoyo a la X11 se introdujo el protocolo. Con el paso de los años, la utilidad de la versión de terminal puede disminuir y, con ella, el interés en mantenerla, porque hacer que emacs se comporte de manera diferente dependiendo de la gráfica / terminal es una carga y un chip de complejidad. Me gustaría ver una abstracción de la GUI de emacs sobre algo como el sixelprotocolo, de modo que las interfaces gráficas y de terminal puedan lograr un nivel más profundo de uso compartido de código. Esto requeriría que emacs invirtiera la suposición de "terminal primero, gráfico segundo" (que en el momento de escribir este artículo creo que todavía se mantiene).[3]

  • El código asincrónico en elisp no es fácil, o es posible en algunos casos. Emacs admite subprocesos que producir es decir, que puede pausar y devolver el control al planificador principal, que es una implementación de multitarea cooperativa . ¿Emacs necesita soporte moderno para primitivas asíncronas? El hecho de que emacs se haya restringido principalmente a un núcleo significa que el código está bastante optimizado, y emacs también puede ejecutarse bastante rápido en dispositivos más pequeños. Creo que emacs, al ser una interfaz interactiva, solo debe admitir una interfaz gráfica asincrónica . En este caso, el alcance y el contexto del código asincrónico se restringirían solo a la interfaz de usuario. Este es el diseño utilizado en otros editores basados ​​en electrones como VSCode , donde la interfaz de usuario se ejecuta en su propio proceso independiente. Particularmente me gustaría para el experiencia de escritura para nunca ser interrumpido, y estoy bien intercambiando el resaltado de sintaxis retrasado, la finalización u otras herramientas auxiliares por suave en tiempo real latencia de entrada.

  • Dado que en emacs se supone que debes tener acceso a todo , la forma en que emacs habla de X11 es bastante codiciosa ya que emacs casi vuelve a implementar un servidor X11 completo dentro de sí mismo. Esto ha causado históricamente algunos problemas.[4] ya que la mayoría de las veces emacs se usa como una ventana dentro de otro X11, y no como una Servidor X11 en sí . El servidor de visualización, que maneja las sesiones gráficas. ¿Cómo sería un entorno de visualización diferente? Imagino algo que comienza desde un nivel inferior y se parece más a un motor de juego, tal vez basado en SDL . Además de la salida de video directa básica, sería accesible desde elisp (compilado de forma nativa) o una API C (o rust) estable. El sistema de ventanas emacs se implementaría en la parte superior del backend "pixel cruncher" y ya no dependería de un conjunto de herramientas gráficas externas como lucid o GTK y lograr un mayor nivel de agnosticismo de plataforma. También significaría un rendimiento más confiable en todos los sistemas, porque cuando la pantalla es manejada por un conjunto de herramientas gráficas externo, el rendimiento puede volverse impredecible y, por ejemplo, cosas como engendrando un nuevo marco terminan siendo más lentos en Windows que en Linux; cuando estas funcionalidades se utilizan con frecuencia, como los candidatos de finalización de pantalla, las discrepancias se hacen evidentes.[5]

  • Tener plataformas de visualización más modernas permitiría no solo un mejor rendimiento, sino también el acceso a herramientas visuales más avanzadas, como transiciones adecuadas, sombras, desenfoque y un mejor sistema de capas, ya que emacs actualmente las implementa con superposiciones que escalan mal. En general, sería bueno tener todo el poder de CSS3 y más, listo para usar y mejorar las interfaces emacs actualmente algo espartanas.[6]

  • Trabaja en alguna especificación para elisp. Los mantenedores actuales de emacs se preocupan por la capacidad de ejecutar emacs en diferentes plataformas (arquitecturas de cpu). Si la mayor parte del núcleo se reescribe en elisp mismo[7], lo único que queda en el interior se convierte en el intérprete elisp. Con un acuerdo sobre alguna forma de especificación elisp, la migración de la totalidad de emacs a un tiempo de ejecución diferente se vuelve factible, ya que el problema es equivalente a "simplemente" implementar un intérprete elisp.

  • ¿Una plataforma de financiación emacs? En esto publicación de reddit se menciona que es difícil "simplemente financiar" a las personas para que trabajen en las cosas, y deben ser los desarrolladores interesados ​​en trabajar en funciones específicas que deben solicitar financiación. Sin embargo, esto no tiene en cuenta el hecho de que no muchas personas conocen bien los aspectos internos de emacs. Efectivamente, esperar a que los desarrolladores obtengan el conocimiento para implementar características específicas puede llevar 20 años (y contando); lo que aceleraría las cosas sería primero Pague a las personas para que aprendan (¡y escriban documentación!) sobre los componentes internos de emacs, de modo que más personas puedan familiarizarse con él y mover lentamente el desarrollo de emacs desde el catedral y bazar modelo de desarrollo hacia uno más inclusivo.

  • Lista de deseos de paquetes:

    • Sentarse en el árbol ¡todas las cosas! El resaltado de sintaxis de Emacs se basa en expresiones regulares, que no son lentas (la mayoría de las veces) pero pueden causar algunos obstáculos en el rendimiento que sería deseable evitar por completo. Pero treesitter no solo ayuda con el resaltado de sintaxis, sino que proporciona una representación jerárquica del contenido del búfer que también beneficia a otros paquetes:

      • smartparens, paredit, lispyville, etc ...: Este es el paquete que proporciona la API más funcional para aplicar manipulaciones de texto basadas en objetos de texto, como slurping, pero son lentas (smartparens en particular). Volver a basar la API en la parte superior de treesitter parece atractivo.

      • Polymode: es un modo menor para mezclar diferentes modos principales en el mismo búfer. Aplica el estrechamiento de regiones a través de expresiones regulares, el uso de treesitter aquí ayudará a acelerar las cosas.

      • Orgmode: después de que se haya escrito una gramática para tree sitter, orgmode podría recibir considerables baches de velocidad, ya que gran parte de la lógica y ejecución del modo org se basa en el contenido del búfer que debe analizarse regularmente (propiedades, bloques de código, encabezados, salidas)

    • Agregar hipérbole a orgmode: este paquete implementa hipervínculos globalmente, mientras que los hipervínculos orgmode solo funcionan entre búferes orgmode. También orgmode, que se basa en el modo de esquema, podría beneficiarse del modo kline proporcionado por la hipérbole, que implementa nodos que no colisionan.

    • Circhat: una combinación de palabras entre circe.el y weechat.el . Circe implementa una muy buena interfaz de usuario para búferes de chat, mientras que weechat.el brinda soporte para interactuar con el protocolo de retransmisión weechat , pero tiene una interfaz de usuario pobre y con errores. El uso de weechat para manejar las comunicaciones por chat libera a emacs de tener que manejar todas las conexiones de IRC y monitorear todos los búferes, lo que puede ralentizar bastante a emacs. Además, weechat tiene soporte de complementos para matrix, por lo que también obtienes soporte de matrix directo dentro de emacs.

    • Tramp: tramp es el paquete emacs que maneja conexiones remotas, pero es terriblemente lento. Esto no es necesariamente culpa de todos los vagabundos, porque otros paquetes nunca comprueban si la sesión actual es local o remota, y aplican cosas como operaciones pesadas del sistema de archivos en puntos finales remotos. Pero tramp también carece de soporte asíncrono, que es un estado de cosas muy triste para algo que en su mayoría envía comandos a procesos externos que manejan los protocolos solicitados (ssh, sftp, docker ...). La experiencia de VSCode con eliminar hosts es mucho mejor, y emacs necesita mejorar su juego :)
