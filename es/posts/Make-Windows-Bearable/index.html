Hacer soportable Windows

Una lista de herramientas para hacer que Windows se sienta más como Linux

Desde que yo cambió a Windows 10 con WSL Pensé que haría una lista del software que tenía que usar para acercar Windows a mi flujo de trabajo habitual de Linux. Dejemos en claro que la interfaz de usuario de Windows apesta, está atascada en el90s y se supone que debe hacer clic y arrastrar todo con el puntero del mouse. No se si es un cultura cosa (a los desarrolladores de Windows les debe gustar mucho el mouse), o para objetivo demográfico ("¡haga clic en esa cosa!" es fácil de explicar), o debido a mejores prácticas de seguridad (si le damos demasiado control al usuario, el malware lo explotará) o simplemente el Shell de Windows estar hecho de una pila de deuda tecnológica que dificulta agregar funcionalidad para usuarios avanzados. El hecho es que las cosas simples en Linux necesitan soluciones complicadas en Windows.

Escritorios virtuales

En Linux, cada administrador de Windows tiene algún tipo de escritorio virtual, Windows se construyó escritorios virtuales recientemente con W10. Lo que pasa es que en el momento de escribir este artículo, solo puede desplazarse hacia la izquierda y hacia la derecha con las teclas de acceso rápido, y no hay teclas de acceso rápido para ir al escritorio X. Existe esto biblioteca vd pero no he encontrado ningún software que lo utilice, solo AHK[1] scripts con implementaciones deficientes como "bucle hasta que estemos en el VD correcto".

La falta de accesos directos adecuados para escritorios virtuales integrados en Windows me obligó a utilizar software de terceros, de los cuales hay muchos, elegí dexpot ya que hace todas las cosas que necesitaría y más (sería feliz con sologo to X ymove window to X atajos). El único problema fue que dexpot no puede vincular claves ya vinculadas ... algo AHK puede anular las claves utilizadas por otras aplicaciones, mientras que dexpot no lo hace, supongo que se debe a que se basa en una API de Windows diferente (o [dll]). La solución es reasignar los accesos directos que quería (p. Ej.Win+1 ) con un atajo gratuito que se puede configurar en dexpot (p. ej.Win+Shift+F1)

Mapeo de teclado

Powertoys permite mapear teclas y atajos, de modo que puedo mapear CapsLock-> LeftControl y RightControl-> LeftControl. Y atajos comoWin+hjkl a flechas. DemasiadoWin+l es un acceso directo predeterminado de Windows para bloquear la pantalla y no se puede reasignar ... una clave de registro anula los accesos directos y corrige esto ... hasta que actualice Windows ... ¡así que debe aplicar la corrección en cada arranque!

Lanzacohetes

En cambio, el lanzador de aplicaciones proporcionado por powertoys es bastante funcional, no usa mucha más memoria que KRunner y es rápido y receptivo. La única queja es que, de vez en cuando, pierde el enfoque, pero lo más probable es que sea un problema con el control enfermo (como en pervertido) de Windows sobre qué ventana se supone que debe enfocarse en un momento dado.

Embaldosado

Cambié de Sway a KDE en Linux, por lo que no era un gran amante de las ventanas en mosaico. Powertoys tiene FancyZones, que es una ligera mejora con respecto a los Snaps básicos de Windows que ofrece la ventana, ya que te brinda brechas y diseños . Sin embargo, todavía falta lo más importante que hace que el mosaico sea útil, es decir normaspara aplicar a las ventanas coincidentes, pero esto también necesitaría soporte para escritorios virtuales ... y ya hemos mencionado su estado actual. En comparación, KWin le permite hacer coincidir ventanas con definiciones muy complejas. FancyZones soluciona el problema de las reglas preservando las posiciones de las ventanas en las sesiones, pero no he investigado qué tan bien funciona esto con diferentes escritorios virtuales.

Arrastrando ventanas

Windows solo se puede arrastrar desde la barra de título y un utilidad debe usarse para moverlos con una combinación de atajos.

Monitor del sistema

Con KDE tienes widgets de plasma para mostrar información del sistema, aunque los widgets de plasma tienden a consumir mucha memoria, así que no los usé tanto. Con Windows no hay una utilidad incorporada para esto, después de probar algunas utilidades, me decidí Monitor de tráfico que proporciona red, cpu, memoria, y recientemente se agregaron GPU y temps. Servicios del sistema

A muchos no les gustó systemd cuando apareció en el kernel de Linux. En cambio, con Windows empecé a perderlo ... Servicios de Windows no es realmente algo orientado a los usuarios como systemd. Windows tiene la Programador de tareas para lidiar con cosas que parecenone-shot unidad de archivos, pero de nuevo su interfaz (o la falta de ella) es terrible. Por suerte, no necesitamos demonios personalizados en Windows, ya que se supone que la mayoría de las aplicaciones de Windows que usamos son solo ejecutar en el arranquetipo de lógica, y no necesita configuraciones más avanzadas. De hecho, aparte de la basura de herramientas ( suspiro ) para que Windows funcione como un entorno moderno, entonces solo otras aplicaciones nativas de Windows en uso son el navegador ( Firefox ) ya que la aceleración de la GPU del navegador es mala dentro de WSL (bueno, incluso en Linux nativo ...) y el reproductor de video ( mpv ) .. y juegos por supuesto ...

Paquetes

No me gusta achocolatado como requiere privilegios de administrador, siempre busco primero los paquetes primarios, ya que están instalados en la carpeta del usuario, lo cual es más conveniente y más consistente, y simplifica las copias de seguridad.

WSL / g

Me indujeron a cambiar a Windows desde el reciente WSLg actualizar. ¿Qué controladores de mesa requeridos compilados con eld3d12 soporte de backend paraopengl . Windows proporciona una capa de ubuntu de vista previa de la comunidad, pero opté por arch linux dado que ya existía un paquete AUR para mesa con d3d12 . Dado que WSL actualmente no es compatible con systemd, terminé usando supervisor para gestionar algunos servicios. Para asegurarse de que el supervisor esté activo, hay una verificación en el perfil de shell para un archivo de bloqueo que debe crearse entmpfs si el supervisor se ha iniciado previamente.

WSL solo admiteext4 sistemas de archivos, para usar otros sistemas de archivos, debe montar la partición (o el disco) directamente dentro de la VM WSL. Sin embargo, deben montarse manualmente. como administrador desde las ventanas. Para automatizar esto, podemos usar el programador de tareas que permite omitir el indicador de UAC ejecutando tareas. con el mayor privilegio . Podemos ejecutar un wsl--mount comando para montar nuestros discos / particiones deseados y luego ejecutar montar el sistema de archivos desde linux. Porque mi/home/ reside en un montadobtrfssistema de archivos tengo que montarlo automáticamente, así que usamos/etc/fstab para mapear nuestra partición (porLABEL ) para/home . Porque usamos nix connix-env tenemos que vincular el montaje de nuestra tienda nix a/nix en cada inicio de WSL, y porque/tmp no está encendidotmpfs tenemos que crear una montura superpuesta que se montetmpfs sobre lo existente/tmp directorio, para preservar archivos importantes, específicamente los archivos X11, necesarios para comunicarse con el servidor X.

Estos montajes se ejecutan con un script, que debe ejecutarse después hemos montado el disco dentro de linux. Necesitamos una tarea que se ejecute después de que se complete la tarea de montaje, y debido a que WSL usa diferentes VM para diferentes usuarios de Windows, el script NO debe ejecutarse con los privilegios más altos (de lo contrario, se montaría en la VM del administrador).

Para garantizar que WSLg funcione, debemos asegurarnos de que el XDG TIEMPO DE FUNCIONAMIENTO DIR está configurado, ya que es diferente y está ubicado (por defecto) en/mnt/wlsg/runtime-dir . Así es como se ven las ventanas en wayland, puede ver que dado que con wayland las decoraciones de las ventanas pueden ser dibujadas por el compositor o la aplicación, mantienen la configuración de su tema GTK / QT, por lo que termina con una ventana de aspecto nativo de Linux. dentro de un shell de Windows ... lo cual es un poco desorientador al principio.

Emulador de terminal

En linux estaba usando gatito , ya que era uno de los emuladores de terminal más nuevos con aceleración de GPU[2] y un modo demonio algo estable que permite múltiples ventanas con la misma instancia. Sin embargo, en Windows, incursioné con algunos terminales basados ​​en Windows como Windows Terminal, fluent terminal, wsltty, conemu, pero finalmente cambié a wezterm . A pesar de ser una terminal más reciente, tiene todas las características que desearía de un emulador de terminal:

  • pestañas y paneles, de modo que no necesite tmux

  • una multiplexación (modo remoto sin cabeza), de modo que no necesita tmux

  • modo de copia de texto con combinaciones de teclas vi, de modo que no necesita tmux

  • selección rápida, un modo de sugerencia para seleccionar rápidamente objetos de texto desde el desplazamiento hacia atrás

  • comandos predeterminados para nuevas pestañas

  • Muy buena personalización de mapas de teclas, aparte del hecho de que no puede usar Shift / Ctrl / Alt Izquierda / Derecha como modificadores diferentes.

      Con Windows también modifiqué un poco mi flujo de trabajo de modo que no necesito una terminal con modo demonio que admita múltiples ventanas separadas, usando una terminal desplegable. Para hacer wezterm desplegable después de probar muchos scripts AHK desplegables, finalmente encontré uno eso funcionó bastante bien. Sin embargo, una mejor alternativa sería Windows-terminal-terremoto , pero he tenido problemas con que se comporte mal con los escritorios virtuales administrados por dexpot, mientras que mitty-quake-console parece ser compatible con los escritorios virtuales dexpot. Lo más probable es que cuando mejore la experiencia de los escritorios virtuales integrados en Windows, cambie a windows-terminal-quake. WezTerm también es multiplataforma, lo que significa que puedo seguir usándolo con una instalación nativa de Linux. Ha sido un hallazgo realmente afortunado para este conmutador de Windows + WSL, ya que Kitty aún no es compatible con Windows, y las otras alternativas de terminal tenían algunas molestias bastante importantes que No pude llegar a un acuerdo con.
