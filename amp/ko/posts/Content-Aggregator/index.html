뭘 긁어?

스크랩할 대상을 결정하는 것은 콘텐츠의 범주에 따라 다릅니다. 범주를 "주제"라고 합니다.

  • 모든 주제에는 키워드 목록이 있습니다.

  • Google 애드워즈에서 가져온 키워드 목록은 파이썬 API

  • 키워드는 주기적으로 라운드 로빈 순서로 여러 검색 엔진에서 쿼리됩니다. 인스턴스에 여러 주제가 있는 경우 사용 가능한 콘텐츠가 적은 주제가 먼저 검색됩니다. 우리가 의존하는 검색을 수행하기 위해 검색어 프록시로. Searx는 라이브러리 친화적이지 않습니다. 주된 용도가 프런트엔드이기 때문에 쿼리를 수행하기 위해 모듈을 초기화하기 위해 올바른 프로세스를 파헤쳐야 했습니다. 스레드 풀을 사용하여 여러 쿼리를 동시에 수행하여 작업 속도를 높이기 위해 프로젝트 전체에서 스레드 풀을 자주 사용합니다.

  • 모든 키워드 검색은 나중에 처리할 수 있도록 저장소에 저장되는 잠재적 콘텐츠 소스(검색 엔진 결과) 목록을 생성합니다.

  • 특정 주제에 대한 새로운 콘텐츠를 찾으려면 먼저 사용 가능한 소스가 있는지 확인하고 그렇지 않으면 키워드 목록에서 새 소스를 생성합니다.

  • 소스는 두 개의 라이브러리를 통해 처리됩니다. 트라필라투라 주요한 것입니다. 실패하면 백업합니다. 거위 . 또한 추가 링크(새 소스로 간주됨)에 대한 피드를 찾으려고 노력합니다. 우리가 사용하는 피드 피드파인더 그러나 rss에 대한 html의 간단한 구문 분석link 태그로도 충분합니다.

  • 우리의 주요 콘텐츠 유형은Article , 파이썬 측에서 그것은 단지 dict이고 nim 측에서는 그것을 객체로 구문 분석합니다. 열쇠:

    • title : 기사의 머리글

    • content : 기사 자체. 좋은 기사를 결정하기 위해 다양한 필터링 단계를 거칩니다.

      • 먼저 trafilatura 또는 goose에 텍스트가 있고 충분히 긴지 확인합니다. 최소 크기는 300단어입니다. 크기가 일치하지 않으면 소스를 버립니다(아무것도 반환하지 않음).

      • 그런 다음 제목을 가져오고 URL과 공백을 제거하여 삭제합니다.

      • 언어가 외국어인 경우 내용과 제목을 모두 영어로 다시 번역합니다(영어를 정규화함).

출판

게시할 콘텐츠가 있으면 무엇을 얼마나 자주 게시할지 결정해야 합니다. 이전에 언급한 것처럼 표시할 항목을 선택하는 것은 사용자에 따라 다르기 때문에 여기에서 어떤 트릭도 생각해내지 않았습니다. 그래서 우리는 최신에서 가장 오래된 , 우리가 더 최근에 스크랩한 것이 더 관련성이 있다는 추론과 함께, 그것은 LIFO 대기줄. 퍼블리싱은 파이썬 도구에 상당히 의존하지만 님에도 있는 서버에 인접하여 실행되기 때문에 님에서 발생합니다.

게시 논리

게시는 지속적으로 발생하며 스크래핑과 마찬가지로 유휴 간격이 있는 것처럼 게시는 스크래핑과 반대로 수행됩니다. 우리가 긁을 때, 우리는 충분히 길다 게시되지 않은 기사의 캐시, 게시와 함께 캐시가 너무 많이 줄어들기 시작하면 속도가 느려집니다. 이런 식으로 스크래핑과 함께 항상 일부 콘텐츠 출판될 언젠가 미래에. 실제 게시 논리:

  • 캐시에서 게시되지 않은 기사 배치를 가져옵니다. (실행당 3개의 새 기사를 게시하도록 선택합니다.)

  • 중복 여부를 확인합니다. Dup 검사는 nim lib를 활용하여 지역에 민감한 해싱을 통해 수행됩니다. 민하쉬 . LSH는 상당히 CPU 집약적이며(알다시피...해싱) 자체 스레드가 필요합니다.

  • 페이지 렌더링: 서버가 쿼리를 즉석에서 처리하므로 필요하지 않지만 여기서 렌더링은 사전 캐싱의 한 형태입니다.

  • 페이지 처리. 우리는 사이트를 다루기 때문에 페이지당 표시할 기사 수를 선택하고 더 많은 기사를 게시함에 따라 페이지를 늘려야 합니다. ~10페이지의 기사를 그룹화하도록 선택합니다. 최신 페이지는 항상 10개 미만의 기사입니다.

  • 게시된 기사의 상태를 저장합니다. 이는 기사를 "게시되지 않음"에서 "게시됨" 상태 및 LSH 데이터베이스로 이동하는 것을 의미합니다.

  • 새 기사를 게시한 후에는 오래된 캐시를 지워야 합니다. 홈페이지, 주제 페이지, 사이트맵, RSS 피드를 지워야 합니다.

const
  rxend = "(?=/+|(?=[?].*)|$)"
  rxAmp = fmt"(/+amp{rxend})"
  rxLang = "(/[a-z]{2}(?:-[A-Z]{2})?" & fmt"{rxend})" # split to avoid formatting regex `{}` usage
  rxTopic = fmt"(/+.*?{rxend})"
  rxPage = fmt"(/+(?:[0-9]+|s|g|feed\.xml|sitemap\.xml){rxend})"

use std::ffi::CStr;

#[no_mangle]
pub extern "C" fn minify(
    code: *const c_char,
    do_not_minify_doctype: bool,
    ensure_spec_compliant_unquoted_attribute_values: bool,
    keep_closing_tags: bool,
    keep_comments: bool,
    keep_html_and_head_opening_tags: bool,
    keep_spaces_between_attributes: bool,
    minify_css: bool,
    minify_js: bool,
    remove_bangs: bool,
    remove_processing_instructions: bool,
) -> *const c_char {

    let code = unsafe { CStr::from_ptr(code) };
    let code_vec = code.to_bytes();

    let cfg = Cfg {
        do_not_minify_doctype,
        ensure_spec_compliant_unquoted_attribute_values,
        keep_closing_tags,
        keep_comments,
        keep_html_and_head_opening_tags,
        keep_spaces_between_attributes,
        minify_css,
        minify_js,
        remove_bangs,
        remove_processing_instructions,
    };

    let minified = minify_html_native(code_vec, &cfg);

    let s = unsafe { CString::from_vec_unchecked(minified).into_raw() };
    return s;

proc pop*[K, V](t: AsyncTable[K, V], k: K): Future[V] {.async.} =
  var popped = false
  withLock(t.lock):
    if k in t.table:
      popped = t.table.pop(k, result)
  if not popped:
    if k notin t.waiters:
      t.waiters[k] = newSeq[ptr Future[V]]()
    var fut = newFuture[V]("AsyncTable.pop")
    t.waiters[k].add fut.addr
    result = await fut

proc put*[K, V](t: AsyncTable[K, V], k: K, v: V) {.async.} =
  withLock(t.lock):
    if k in t.waiters:
      var ws: seq[ptr Future[V]]
      doassert t.waiters.pop(k, ws)
      while ws.len > 0:
        let w = ws.pop()
        if not w.isnil and not w[].isnil and not w[].finished:
          w[].complete(v)
    else:
      t.table[k] = v

proc requestTask(q: sink ptr Request) {.async.} =
  var trial = 0
  var
    sess: HttpSessionRef
    req: HttpClientRequestRef
    resp: HttpClientResponseRef
    cleanup: seq[Future[void]]
  while trial < q[].retries:
    try:
      trial.inc
      sess = new(HttpSessionRef,
                proxyTimeout = 10.seconds.div(3),
                headersTimeout = 10.seconds.div(2),
                connectTimeout = 10.seconds,
                proxy = if q[].proxied: selectProxy(trial) else: "",
                flags = if q[].proxied: proxiedFlags else: sessionFlags
      )
      req = new(HttpClientRequestRef,
                sess,
                sess.getAddress(q[].url).get,
                q[].meth,
                headers = q[].headers.toHeaderTuple,
                body = q[].body.tobytes,
        )
      resp = await req.fetch(followRedirects = q[].redir, raw = true)
      checkNil(resp):
        defer:
          cleanup.add resp.closeWait()
          resp = nil
        q.response.code = httpcore.HttpCode(resp.status)
        checkNil(resp.connection):
          q.response.body = bytesToString (await resp.getBodyBytes)
          q.response.headers = newHttpHeaders(cast[seq[(string, string)]](
              resp.headers.toList))
        break
    except:
      cdebug():
        logexc()
        debug "cronhttp: request failed"
    finally:
      if not req.isnil:
        cleanup.add req.closeWait()
      if not resp.isnil:
        cleanup.add resp.closeWait()
      if not sess.isnil:
        cleanup.add sess.closeWait()
  httpOut[q] = true
  await allFutures(cleanup)

결론

악마가 세부 사항에 있기 때문에 언급하지 않은 많은 것들이 있습니다...그러나 이것은 다음과 같은 전체 코드 기반의 대략적인 둘러보기입니다.

  • 님의 ~12k 라인

  • ~400줄의 js

  • ~1000줄의 scss

  • ~3500줄의 파이썬

  • 74 lines of rust (for bindings :P)

나는 무엇을 다르게 할 것인가?

  • 아마도 전체를 녹슬게 다시 작성할 것입니다. 님은 현재 메모리 안전을 제대로 처리하지 못하고 충돌을 수정하기 위해 gdb에 의존해야 하는 시간이 너무 많았고 모든 문제를 수정하지도 못했습니다. 생태계의 절반이 GC에 의존하고 나머지 절반이 ORC(또는 오크가 아닌 ARC만)에 의존하는 것은 큰 문제입니다. 비동기와 스레드를 혼합하는 것도 고통스럽고 비동기 스택 추적은 악몽입니다. ​​(이 점에서 녹이 더 나은지는 모르겠습니다.)

  • 대상 a PWA getgo에서. 이 프로젝트는 상당히 골치아픈 시작을 했습니다. 처음에는 웹 서버에서 제공하는 정적 페이지로 간주되었지만 웹 서버 자체가 되었습니다. 상호작용은 나중에 생각하게 되어 렌더링된 html과 js/css의 혼합이 되었습니다. 이로 인해 아무런 구조도 없이 나온 API에 대해 너무 느슨해졌습니다(완전히 불안정한 ). 재작성에서는 UI 프레임워크를 사용합니다. 미리 AMP를 완벽하게 지원하거나 solidjs.

  • 인기 있는 플랫폼을 위한 임시 파서를 더 추가하십시오. 일반 기사 구문 분석은 오늘날 가장 인기 있는 플랫폼이 매우 적은 콘텐츠, 많은 비디오 및 이미지를 제공할 때 제대로 작동하지 않거나 전혀 작동하지 않으므로 스크래핑은 텍스트가 아닌 리치 미디어를 더 대상으로 해야 합니다. 스크레이퍼 아키텍처를 계획할 때 고려하면 APP가 제공할 콘텐츠와 정보의 균형이 맞지 않습니다.
