Gör Windows uthärdligt

En lista med verktyg för att få Windows att känna sig mer som Linux

Sedan jag bytte till Windows 10 med WSL Jag tänkte att jag skulle göra en lista över programvara som jag måste använda för att föra Windows närmare mitt vanliga linux -arbetsflöde. Låt oss göra det klart att Windows UI suger, det sitter fast i90s och allt ska klickas och dras med muspekaren. Jag vet inte om det är en kultur sak (windows devs måste tycka mycket om musen), eller för mål demografisk ("klicka på den saken!" är lätt att förklara), eller på grund av säkerhets bästa praxis (om vi ger för mycket kontroll till användaren kommer malware att utnyttja den) eller bara Windows Shell består av en hög med tekniska skulder som gör det svårare att lägga till funktionalitet för avancerade användare. Faktum är att enkla saker i Linux behöver invecklade lösningar i Windows.

Virtuella skrivbord

I Linux har alla Windows -hanterare någon form av virtuellt skrivbord, Windows har nyligen byggt virtuella skrivbord med W10. Saken är att du i skrivande stund bara kan cykla åt vänster och höger med snabbtangenter, och det finns inga snabbtangenter för att gå till skrivbordet X. Det finns det här vd -bibliotek men jag har inte hittat programvara som använder den, istället bara AHK[1] skript med dåliga implementeringar som "loop tills vi är på rätt VD".

Avsaknaden av korrekta genvägar för Windows inbyggda virtuella skrivbord fick mig att tvinga till att använda programvara från tredje part, av vilka det finns många, jag valde dexpot eftersom det gör alla saker jag skulle behöva och mer (jag skulle vara nöjd med barago to X ochmove window to X genvägar). Det enda problemet var att dexpot inte kan binda redan bundna nycklar ... något AHK kan åsidosätta nycklar som används av andra applikationer, medan dexpot inte gör det, jag antar att det beror på att det är beroende av ett annat Windows API (eller [dll]). Lösning är att göra om genvägar till de jag ville (t.ex.Win+1 ) med en gratis genväg som kan ställas in på dexpot (t.ex.Win+Shift+F1)

Tangentbordskartläggning

Powertoys tillåter att kartlägga nycklar och genvägar så att jag kan kartlägga CapsLock-> LeftControl och RightControl-> LeftControl. Och genvägar somWin+hjkl till pilar. SyndWin+l är en Windows standardgenväg för att låsa skärmen och kan inte återskapas ... en registernyckel åsidosätter genvägarna och fixar detta ... tills du uppdaterar windows ... så du måste tillämpa korrigeringen vid varje start!

Launcher

Appstarteren som tillhandahålls av powertoys är istället ganska bra, använder inte mycket mer minne än KRunner och är snabb och lyhörd. Det enda klagomålet är att det ibland tappar fokus, men det är troligtvis ett problem med Windows sjuk (som i pervers) kontroll över vilket fönster som ska vara fokuserat vid en viss tidpunkt.

Kakel

Jag bytte från sway till KDE på Linux, så jag var inte så mycket av en kaklat fönsterälskare. Powertoys har FancyZones vilket är en liten förbättring jämfört med grundläggande fönster Snaps som fönstret erbjuder, som det ger dig luckor och layouter. Ändå saknas det fortfarande det viktigaste som gör kakelplattor användbara, det vill säga regler att applicera på matchande fönster, men detta skulle också behöva stöd för virtuella skrivbord ... och vi har redan nämnt deras nuvarande tillstånd. I jämförelse KWin låter dig matcha fönster med mycket komplexa definitioner. FancyZones arbetar kring reglerfrågan genom att bevara Windows -positioner över sessioner, men jag har inte undersökt hur bra det fungerar med olika virtuella skrivbord.

Dra fönster

Windows kan bara dras från titelfältet och ytterligare ett verktyg måste användas för att flytta dem med en genvägskombination.

Systemmonitor

Med KDE har du plasma -widgets för att visa systeminformation, även om plasma -widgets tenderar att konsumera mycket minne så jag använde dem inte så mycket. Med Windows finns det inget inbyggt verktyg för detta, efter att ha provat några verktyg jag bestämde mig för Traffic Monitor som ger nätverk, cpu, minne och nyligen tillkom GPU och temps. Systemtjänster

Många gillade inte systemd när det kom in i Linux -kärnan..Med Windows började jag istället sakna det..Windows Services är egentligen inte något inriktat på användare som systemd. Windows har Schemaläggaren att hantera saker som ser utone-shotenhetsfiler, men återigen är dess gränssnitt (eller brist på det) fruktansvärt. Tur att vi inte riktigt behöver anpassade demoner i Windows, eftersom de flesta Windows -program vi använder är tänkta bara Körs vid uppstart typ av logik, och behöver inte mer avancerade konfigurationer. I själva verket, bortsett från kullen med verktyg ( suck ) för att få Windows att fungera som en modern miljö, då är det bara andra inbyggda Windows -program som används som webbläsare ( firefox ) eftersom webbläsarens GPU -acceleration är dålig inom WSL (ja även på native linux ...) och videospelaren ( mpv ) .. och spel förstås ...

Paket

Jag gillar inte chokladaktig eftersom det kräver administratörsbehörighet letar jag alltid först efter scoop -paket eftersom de installeras i användarmappen, vilket är mer bekvämt och mer konsekvent och förenklar säkerhetskopiering.

WSL/g

Jag fick mig att byta till fönster från det senaste WSLg uppdatering. Vilket krävde mesa -drivrutiner kompilerade medd3d12 backend -stöd föropengl . Windows ger ett ubuntu -lager för förhandsvisning av communityn, men jag valde det arch linux eftersom det redan fanns ett AUR -paket för mesa med d3d12 . Eftersom WSL för närvarande inte stöder systemd, slutade jag använda handledare att hantera några tjänster. För att säkerställa att handledaren är aktiv kontrolleras i skalprofilen för en låsfil som ska skapas påtmpfs om handledare har startats tidigare.

WSL stöder endastext4 filsystem, för att använda andra filsystem måste du montera partitionen (eller disken) direkt i WSL VM. De måste dock monteras manuellt som administratörfrån fönster. För att automatisera detta kan vi använda uppgiftschemaläggaren som gör det möjligt att kringgå UAC -prompten genom att köra uppgifter med den högsta förmånen . Vi kan köra en wsl--mount kommando för att montera våra önskade skivor/partitioner och sedan köra montera filsystemet från Linux. Eftersom min/home/ bor i en monteradbtrfs filsystemet Jag måste montera det automatiskt, så vi använder/etc/fstab för att kartlägga vår partition (avLABEL ) till/home . Eftersom vi använder nix mednix-env vi måste binda mount vår nix store till/nix vid varje wsl -start, och eftersom/tmp är inte påtmpfs vi måste skapa ett överlagringsfäste som monterastmpfs över det befintliga/tmp katalog, för att bevara viktiga filer, särskilt X11 -filerna, som krävs för att prata med X -servern.

Dessa fästen körs med ett skript, som ska köras efter vi har monterat disken inuti linux. Vi behöver en uppgift som körs efter att monteringsuppgiften har slutförts, och eftersom WSL använder olika virtuella datorer för olika Windows -användare måste skriptet INTE köras med högsta behörighet (annars skulle det kunna monteras på administratörens virtuella dator).

För att säkerställa att WSLg fungerar måste vi se till att XDG KÖRNING DIR är inställt, eftersom det är annorlunda och ligger (som standard) vid/mnt/wlsg/runtime-dir . Så här ser fönster under landvägen ut, du kan se att eftersom med fönsterdekorationer i vägmark kan ritas av antingen kompositören eller applikationen, behåller de konfigurationen av ditt GTK/QT -tema, så att du får ett inbyggt linux -temafönster inuti ett Windows -skal ... vilket är lite desorienterande först.

Terminalemulator

På linux använde jag pott , eftersom det var en av de nyare terminalemulatorerna med GPU -acceleration[2]och ett något stabilt daemon -läge som tillåter flera fönster med samma instans. Men på windows, jag dabbled med några Windows -baserade terminaler som Windows Terminal, flytande terminal, wsltty, conemu, men så småningom bytte jag till wezterm . Trots att den är en nyare terminal har den alla funktioner du vill ha från en terminalemulator:

  • flikar och rutor, så att du inte behöver tmux

  • en multiplexing (fjärrläge utan huvud), så att du inte behöver tmux

  • text kopieringsläge med vi nyckelbindningar, så att du inte behöver tmux

  • snabbval, ett tipsläge för att snabbt välja textobjekt från rullning tillbaka

  • standardkommandon för nya flikar

  • Mycket bra anpassning av nyckelkartor, förutom det faktum att du inte kan använda vänster/höger Shift/Ctrl/Alt som olika modifierare.

Med Windows har jag också ändrat mitt arbetsflöde lite så att jag inte behöver en terminal med daemon -läge som stöder flera separata fönster, med hjälp av en nedrullningsterminal. Att göra wezterm falla ner efter att ha provat många nedrullningsbara AHK-skript, så småningom hittade jag ett som fungerade ganska bra. Ett bättre alternativ skulle dock vara windows-terminal-quake, men jag har haft problem med att det missuppträdde med de virtuella skrivborden som hanteras av dexpot, medan mitty-quake-konsol verkar vara kompatibel med dexpot virtuella skrivbord. Mest troligt när den inbyggda Windows virtuella stationära upplevelsen förbättras kommer jag att byta till windows-terminal-quake. WezTerm är också plattformsoberoende vilket innebär att jag kan fortsätta använda den med en inbyggd Linux-installation. Det har varit en riktigt lycklig upptäckt för denna Windows+WSL -switch, eftersom Kitty ännu inte stöder Windows, och de andra terminalalternativen, hade några ganska stora irritationer som Jag kunde inte komma överens med.
