Emacs

为什么选择 emacs，以及我个人的 emacs 愿望清单。

emacs 是什么？

一种 虚拟机 或口译员 电子书 lisp 编程语言......有点。 emacs lisp 编程语言 (elisp) 是通用的，但对运行它的实际文本编辑器具有一流的支持。原始类型专注于文本编辑，但是你可以用漂亮的方式编写任何你想要的东西 体面的表现 （最近感谢 jit 编译 ) 因为它是 lisp 的后代 MACLISP 和一个兄弟姐妹 常见的口齿不清 .然而，emacs lisp 没有标准，该规范相当于最流行的实现，即 GNU Emacs。

为什么是 emacs？

“Emacs 是一个操作系统”这句话的意思是你可以用它来做任何你可以用电脑来做的事情。由于内核（通常是 linux）仍然负责硬件，因此它不能成为真正的操作系统。[1] 然而，它可以很容易地称为桌面环境，因为它提供了一个工作环境。它使您能够编写快速的交互式代码以与任何应用程序交互，它可以像 shell 脚本一样脏，或者经过深思熟虑以提供稳定的 API。事实上，许多 emacs 功能都是由包提供的。[2] 其他编辑器可以像 emacs 一样可扩展吗？没有为什么？因为其他编辑器使用不同的用户模型，用户必须受到限制，因为它被视为向用户提供“服务”的运行环境的“客人”。这不是 emacs 的情况，其中环境和用户成为一体。在 emacs 中，您可以访问所有内容，并且可以通过创建这些内容的相同方式（又名 elisp）修改几乎所有内容。

这值得么？

收益是累积的，是投资，和任何投资一样，你应该期望收益与你投入的多少成正比，因此你必须花一些时间来学习和练习如何使用它。

我的 GNU emacs 愿望清单

Emacs 是一个很好的操作系统，但是一个糟糕的文本编辑器。

...或类似的规定。 GNU Emacs 实现，就像 emacs 一样，起源于八十年代，其核心的许多部分都显示出它们的年龄......

  • 图形界面和终端界面的二分法。由于 emacs 很老，它开始作为在终端内运行的应用程序，但随着 90 年代图形界面的出现，它开始收集更适合于 基于窗口 2000 年左右 X11 引入了协议。随着时间的流逝，终端版本的实用性可能会降低，随之而来的是维护它的兴趣，因为根据图形/终端的不同，emacs 的行为会有所不同，这是一个相当大的负担，而且是一个复杂的芯片。我想在类似的东西上看到 emacs GUI 的抽象 六边形协议，使得图形界面和终端界面可以实现更深层次的代码共享。这将需要 emacs 反转“终端第一，图形第二”的假设（我认为在撰写本文时仍然成立）。[3]

  • elisp 中的异步代码并不容易，或者在某些情况下是可能的。 Emacs 支持的线程 屈服 即可以暂停并将控制权交还给主调度程序，这是 多任务协作 . emacs 是否需要对异步原语的现代支持？ emacs 大多被限制在一个内核上这一事实意味着代码非常优化，而且 emacs 本身也可以在较小的设备上运行得相当快。我认为 emacs 作为一个交互界面，只需要支持一个 异步图形界面 .在这种情况下，异步代码的范围和上下文将仅限于 UI。这是其他基于电子的编辑器中使用的设计，例如 VS代码 ，其中 UI 在其自己的单独进程中运行。我特别想要 打字经验 永远不会被打断，我很好地权衡延迟的语法突出显示、完成或其他辅助工具 软实时 输入延迟。

  • 因为在 emacs 中你应该可以访问 一切 ，emacs 说 X11 的方式非常贪婪，因为 emacs 几乎在其内部重新实现了一个完整的 X11 服务器。这在历史上造成了一些麻烦[4] 因为大多数时候 emacs 被用作另一个 X11 内的窗口，而不是一个 X11服务器本身 .显示服务器，处理图形会话。不同的显示环境会是什么样子？我想象一些从较低层次开始的东西，看起来更像一个游戏引擎，也许基于 SDL .在基本的直接视频输出之上，可以从（本地编译的）elisp 或稳定的 C（或 rust）API 访问。 emacs 窗口系统将在“pixel cruncher”后端之上实现，不再依赖于像 lucid 或 GTK 并实现更高水平的平台不可知论。这也意味着跨系统的性能更可靠，因为当显示由外部图形工具包处理时，性能可能变得不可预测，例如 生成一个新框架 最终在 Windows 上比在 linux 上慢；当这些功能变得频繁使用时，比如显示完成候选，差异就会变得明显。[5]

  • 拥有更现代的显示平台不仅可以提供更好的性能，还可以访问更高级的视觉工具，例如适当的过渡、阴影、模糊和更好的图层系统，因为 emacs 目前实现了它们 覆盖 哪个规模不好。一般来说，拥有 CSS3 及更高版本的全部功能，准备使用和改进目前有些简陋的 emacs 界面会很好。[6]

  • 为 elisp 制定一些规范。当前的 emacs 维护者关心跨不同平台（cpu 架构）运行 emacs 的能力。如果在 elisp 本身中重写了尽可能多的核心[7]，内部唯一剩下的就是elisp解释器。通过就某种形式的 elisp 规范达成一致，将整个 emacs 移植到不同的运行时变得可行，因为该问题相当于“只是”实现了一个 elisp 解释器。

  • emacs 融资平台？在这 Reddit 帖子 提到了“仅仅资助”人们从事某项工作是多么困难，而且必须是对开发特定功能感兴趣的开发人员必须请求资金。然而，这并没有考虑到没有多少人精通 emacs 内部的事实。实际上，等待开发人员获得实现特定功能的知识可能需要 20 年（并且还在增加）；什么会加快速度 第一的 付钱让人们学习（并编写文档！）有关 emacs 内部的知识，以便更多的人可以熟悉它并慢慢地将 emacs 开发从 大教堂和集市 发展模式更具包容性。

  • 套餐愿望清单：

    • 树坐 所有的事情！ Emacs 语法高亮基于正则表达式，它并不慢（大多数情况下），但会导致一些性能悬崖，这是非常希望完全避免的。但是 treeitter 不仅有助于语法突出显示，它还提供了缓冲区内容的分层表示，这也有利于其他包：

      • smartparens、paredit、lispyville 等...：这是一个包，它提供了功能最强大的 api，用于应用基于文本对象的文本操作，例如 slurping，但它们很慢（尤其是 smartparens）。在 treeitter 之上重新构建 api 看起来很有吸引力。

      • Polymode：是一种次要模式，用于在同一缓冲区中混合不同的主要模式。它通过正则表达式应用区域缩小，在这里使用 treeitter 将有助于加快速度。

      • Orgmode：在编写了 tree sitter 的语法之后，orgmode 可能会收到相当大的减速，因为许多 org mode 逻辑和执行基于必须定期解析的缓冲区内容（属性、代码块、标题、输出）

    • 添加 夸张 to orgmode：这个包实现了全局超链接，而 orgmode 超链接只在 orgmode 缓冲区之间工作。此外，基于轮廓模式的 orgmode 可以从实现非碰撞节点的 hyperbole 提供的 kline 模式中受益。

    • Circhat：之间的一个词混搭 圆环 和 微信 . Circe 为聊天缓冲区实现了一个非常好的 UI，而 weechat.el 提供了与 微信中继协议 ，但有一个糟糕和错误的用户界面。使用 weechat 处理聊天通信使 emacs 不必处理所有 IRC 连接和监视所有缓冲区，这会大大降低 emacs 的速度。此外，weechat 有对矩阵的插件支持，因此您也可以在 emacs 中获得直接的矩阵支持。

    • Tramp：tramp 是处理远程连接的 emacs 包，但速度非常慢。这不一定是所有的tramps错误，因为其他包从不检查当前会话是本地还是远程，并在远程端点上应用诸如文件系统繁重操作之类的事情。但是 Tramp 也缺乏异步支持，对于那些主要向处理请求的协议（ssh、sftp、docker...）的外部进程发出命令的东西来说，这是一种非常可悲的状态。删除主机的 VSCode 体验要好得多，而且 emacs 需要加强它的游戏 :)
