为了乐趣和利润而构建内容聚合器？

一个完整的应用程序，可以从网络上抓取、处理和呈现内容……在网络上。

为什么？

信息超载 ？这些天有点糟糕，很多低信噪比的信息源，缩小你的“提要”以免你不知所措有点困难。一种过滤信息并以既容易又快速消化的格式呈现信息的工具将非常有用。这就是为什么我认为 内容聚合 一个常青的破坏领域。它现在是并将永远是（只要互联网是免费的并且有言论自由）一个很好的商业机会。这是其中一个完全与执行有关的实例（与想法无关）。

管理期望

话虽如此，我的应用程序最终没有进行任何实际过滤。事实上它只是 聚集体 内容来自网络。这是因为我没有将用户构建到其中，如果不能为每个用户量身定制，就没有什么动力进行过滤。

决定抓取什么取决于内容的类别。我们称类别为“主题”。

  • 每个主题都有一个关键字列表。

  • 如果使用他们的谷歌广告词提取关键字列表 蟒蛇API

  • 关键字在多个搜索引擎上以循环顺序定期查询。如果实例有多个主题，则首先搜索可用内容较少的主题。执行我们依赖的搜索 搜索 与代理。 Searx 不是完全库友好的，因为它的主要用途是它的前端，所以它需要挖掘正确的过程来初始化模块以执行查询。为了加快使用线程池并发执行多个查询的速度，我们将在整个项目中频繁使用线程池。

  • 每个关键字搜索都会生成一个潜在内容源列表（搜索引擎结果），这些内容源保存在存储中供以后处理。

  • 当我们想要为特定主题查找新内容时，我们首先检查是否有可用资源，否则我们从关键字列表生成新资源。

  • 源通过两个库处理 trafilatura 是主要的，如果失败我们备份 鹅 .我们还尝试寻找额外链接的提要（这将被视为新来源）。对于我们使用的提要 进纸器 但是对 rss 的 html 进行简单解析link 标签也足够了。

      • 如果相关性测试已通过，作为最后一步，我们会清除内容中出现的太多括号、空格、重复字符和特殊字符。

      • 如果清理没有删除所有内容，我们将继续处理文章。

    • author : 作者，否则为源链接首页的标题

    • pubDate ：文章的发表日期，或者现在

    • topic : 这篇文章所属的主题

    • tags : 一篇文章的相关关键词，我们使用最快的kw提取库，即 耙 , 考虑的备选方案是 派特 （组合）， 文本等级 和 短语机器

    • imageOrigin ：如果源解析（对于我们使用的图像 恋人 ) 没有找到图像，我们查询搜索引擎以获取相关图像，因此 imageOrigin 指向托管图像的原始页面，否则它等于源 url。

出版

一旦我们有了要发布的内容，我们就必须决定发布什么内容以及发布频率。我在这里没有想出任何技巧，因为正如我之前提到的，选择要显示的内容取决于用户。所以我们只发布 从最新到最旧 ，因为我们最近抓取的东西更相关，所以它是 后进先出法 队列。发布虽然非常依赖 python 工具，但发生在 nim 中，因为它与服务器相邻运行，而服务器也在 nim 中。

发布逻辑

发布是连续发生的，就像抓取有空闲时间间隔一样，发布也是如此，与抓取相反。当我们刮擦时，我们会放慢速度 够长了 未发表文章的缓存，当我们的缓存开始缩小太多时，我们会放慢发布速度。这样，再加上刮，应该总是有 一些内容 待发表 一段时间 在将来。实际发布逻辑：

  • 从缓存中获取一批未发表的文章。 （我们选择每次运行发表 3 篇新文章。）

  • 检查它们是否重复。重复检查是通过局部敏感散列来完成的，利用 nim 库， minhash . LSH 非常占用 CPU（您知道……散列），并且需要自己的线程（我们处理的其他一些任务需要自己的线程）。

  • 页面呈现：这不是必需的，因为服务器会即时处理查询，但此处呈现是一种预缓存形式。

  • 页面处理。由于我们正在处理一个站点，因此我们必须选择每页显示多少篇文章，并在发布更多文章时增加页面。我们选择以 ~10 页为单位对文章进行分组。最新的页面总是少于 10 篇文章。

  • 保存已发布文章的状态，这意味着将文章从“未发布”状态移动到“已发布”状态，以及 LSH 数据库。

  • 发布新文章后，我们必须清除陈旧的缓存。我们必须清除主页、主题页面以及站点地图和 RSS 提要。

结论

有很多事情我没有提到，因为细节决定成败……然而，这是对整个代码库的粗略浏览，相当于：

  • ~12k 行 nim

  • ~400 行 js

  • ~1000 行 scs

  • ~3500 行 python

  • 74 lines of rust (for bindings :P)

我会做些什么不同的事情？

  • 可能用 rust 重写了整个东西，nim 目前不能很好地处理内存安全，而且我不得不依靠 gdb 来修复崩溃的时间太多了，我什至还没有设法修复所有的崩溃。当一半的生态系统依赖于 GC 而另一半依赖于 ORC（或者甚至不是 orc 而只是 ARC）时，这是一个大问题。混合异步和线程也很痛苦，异步堆栈跟踪是一场噩梦，（尽管我不知道 Rust 在这方面是否更好。）

  • 目标一个 PWA 从一开始。该项目有一个相当麻烦的开始。一开始它应该是由网络服务器提供的静态页面，然后它自己变成了网络服务器。交互性是事后才想到的，所以它只是渲染的 html 和 js/css 的混合体。这让我对 API 太松懈了，它没有任何结构（完全 不平静 ).在重写中，我会使用 ui 框架，要么 预演 具有完整的 AMP 支持，或 solidjs.

  • 为流行平台添加更多临时解析器。当当今最流行的平台提供的内容很少，而视频和图像很多时，普通文章解析效果不佳（或根本不适用），因此抓取必须更多地针对富媒体而不仅仅是文本，如果不是这样的话在规划爬虫架构时考虑到，APP将服务的内容和信息将是不平衡的。
