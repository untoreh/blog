Emacs

Napa emak, lan dhaptar harapan emac pribadi.

Apa emacs?

A mesin virtual utawa juru kanggo emacs lisp bahasa pamrograman ... jenis. Bahasa pemrograman emacs lisp (elisp) minangka tujuan umum, nanging dhukungan kelas kapisan kanggo editor teks nyata sing mbukak. Jinis primitif fokus ing suntingan teks, nanging sampeyan bisa nulis apa wae sing dikarepake kinerja prayoga (bubar matur nuwun kanggo kompilasi jit ) amarga keturunan lisp saka MACLISP lan sedulur saka lisp umum . Nanging lisp emacs ora duwe standar, lan spesifikasi kasebut padha karo implementasine sing paling populer, yaiku GNU Emacs.

Napa emak?

Kanthi tembung "Emacs minangka sistem operasi" wong tegese sampeyan bisa nggunakake kanggo nindakake samubarang sing nggunakake komputer. Ora bisa dadi sistem operasi sing nyata amarga kernel, (biasane linux) isih ngatasi hardware kasebut.[1] Nanging, bisa uga diarani lingkungan desktop amarga nyedhiyakake lingkungan sing bisa digunakake. Sampeyan menehi sampeyan kemampuan nulis kode interaktif cepet kanggo antarmuka karo aplikasi apa wae, bisa dadi reged kaya skrip shell utawa bisa uga dipikirake kanggo nyedhiyakake API sing stabil. Nyatane akeh fungsionalitas emac sing disedhiyakake dening paket.[2] Apa editor liyane bisa ditambahi kaya para emac? Ora. Amarga editor liyane nggunakake model pangguna sing beda, sing pangguna kudu diwatesi amarga dianggep "tamu" lingkungan sing mbukak sing nawakake "layanan" kanggo pangguna. Iki ora beda karo emac, ing endi lingkungan lan pangguna dadi siji lan padha. Ing emacs, sampeyan duwe akses menyang kabeh, lan bisa ngowahi meh kabeh perkara kanthi cara sing padha digawe, alias elisp.

Apa regane?

Mupangate entuk, yaiku investasi, lan kaya investasi apa wae, sampeyan kudu ngarepake bathi sing sebanding karo pinten-pinten sing sampeyan lebokake, mula sampeyan kudu nglampahi sawetara wektu kanggo sinau lan latihan babagan cara nggunakake.

Wishlist GNU saya

Emacs minangka sistem operasi sing apik banget, nanging editor teks sing ala.

... Utawa ana prekara kasebut. Implementasine GNU Emacs, kayadene emacs, diwiwiti ing taun wolung puluhan, lan akeh bagean inti nuduhake umur ...

  • Dikothomi antarane antarmuka grafis lan antarmuka terminal. Wiwit emacs wis tuwa, mula dadi aplikasi kanggo mbukak ing njero terminal, nanging nalika ana antarmuka grafis ing sangang puluhan puluhan taun, mula nglumpukake fungsi sing luwih cocog adhedhasar jendhela interaksi lan udakara taun 2000 dhukungan kanggo X11 protokol dikenalake Sawise pirang-pirang taun, migunani versi terminal bisa uga saya suda, lan minat kanggo njaga, amarga duwe permaisuri tumindak beda-beda gumantung karo grafis / terminal cukup momotan, lan chip kompleksitas. Aku pengin ndeleng abstraksi saka emacs GUI babagan prekara kasebut enemprotokol, saengga antarmuka grafis lan terminal bisa entuk level bareng kode sing luwih jero. Iki mbutuhake emac kanggo mbalikke asumsi "terminal dhisik, grafis liya" (sing nalika aku nulis aku isih duwe).[3]

  • Kode Async ing elisp ora gampang, utawa bisa uga bisa ditindakake. Emacs ndhukung Utas sing ngasilake yaiku sing bisa ngaso lan menehi kontrol maneh menyang panjadwal utama, yaiku implementasi multitasking koperasi . Apa emak butuh dhukungan modern kanggo primitif async? Kasunyatan manawa emac biasane dikepengini dadi siji inti tegese kode kasebut cukup dioptimalake, lan emacs bisa uga mlaku kanthi cepet ing piranti sing luwih cilik. Aku mikir manawa emacs, minangka antarmuka interaktif, mung kudu ndhukung antarmuka grafis asinkron . Ngono, ruang lingkup lan konteks kode asinkronisasi bakal diwatesi mung kanggo UI. Iki desain sing digunakake ing editor adhedhasar elektron liyane kaya VSCode , ing endi UI nganggo proses dhewe-dhewe. Aku pengin pengalaman ngetik supaya ora bisa ngganggu, lan aku dagang kanthi cepet nalika nyorot sintaks, ngrampungake, utawa alat tambahan liyane realtime alus latency input.

  • Wiwit ing emacs sampeyan mesthine duwe akses menyang kabeh , cara emac ngomong X11 cukup rakus amarga emac meh ngetrapake maneh server X11 kanthi lengkap. Iki nyebabake historis sawetara masalah[4] amarga umume kaping emacs digunakake minangka jendhela ing X11 liyane, lan dudu Server X11 dhewe . Server tampilan, sing nangani sesi grafis. Apa tampilan lingkungan tampilan sing beda? Aku mbayangake babagan sing diwiwiti saka level sing luwih murah, lan katon kaya mesin game, bisa uga adhedhasar SDL . Ing ndhuwur output video langsung dhasar, sing bakal bisa diakses saka elisp (sing disusun kanthi asli) utawa API C (utawa karat) sing stabil. Sistem jendela emacs bakal dileksanakake ing ndhuwur backend "pixel cruncher" lan ora ngandelake maneh karo toolkit grafis eksternal kaya lucid utawa GTK lan entuk level agnostisisme platform sing luwih dhuwur. Iki uga tegese kinerja sing luwih dipercaya ing saindenging sistem, amarga nalika tampilan ditangani dening kinerja toolkit grafis eksternal bisa dadi ora bisa ditebak, lan contone kaya pemijahan pigura anyar pungkasane luwih alon ing windows tinimbang ing linux; nalika fungsi kasebut dadi asring digunakake, kayata calon kandidat tampilan, bedane bisa kabukten.[5]

  • Duwe platform tampilan sing luwih modern ora mung nggawe kinerja sing luwih apik, nanging akses menyang alat visual sing luwih maju, kayata transisi, bayangan, kabur, lan sistem lapisan sing luwih apik, amarga para emac saiki nggunakake overlay sing skala ora apik. Umumé, luwih becik yen duwe kekuwatan CSS3 lan liya-liyane, siyap digunakake lan ngapikake antarmuka emacs sing saiki wis spartan.[6]

  • Nggunakake sawetara spesifikasi kanggo elisp. Perawat emak saiki peduli babagan kemampuan mbukak emac ing macem-macem platform (arsitektur cpu). Yen inti minangka inti ditulis maneh ing elisp[7], siji-sijine perkara internal sing dadi interpreter utama. Kanthi persetujuan babagan sawetara jinis spesifikasi penting, portasi kabeh emac menyang macem-macem runtime bisa ditindakake, amarga masalahe padha karo "mung" ngleksanakake interpreter elisp.

  • Platform pendanaan emacs? Ing iki kirim reddit disebutake yen angel "mung dana" wong kanggo ngrampungake, lan kudu dadi pangembang sing kepengin ngupayakake fitur-fitur tartamtu sing kudu njaluk dana. Nanging, iki ora nganggep kasunyatan manawa ora akeh wong sing ahli ing babagan emak internal. Efektif, ngenteni pangembang kanggo entuk ilmu kanggo ngetrapake fitur tartamtu bisa mbutuhake 20 taun (lan ngetung); apa sing bakal nyepetake kabeh dhisik mbayar wong kanggo sinau (lan nulis dokumentasi!) babagan internal emac supaya luwih akeh wong sing bisa ngerti lan alon-alon mindhah pangembangan emacs saka katedral lan bazaar model pangembangan nganti luwih kalebu.

  • Dhaptar paket:

    • Wit-witan kabeh prekara! Nyoroti sintaks Emacs adhedhasar ekspresi reguler, sing ora alon (umume) nanging bisa nyebabake sawetara kinerja kinerja sing bisa diindhari kabeh. Nanging treeitter ora mung mbantu nyorot sintaksis, nanging uga menehi representasi hirarkis konten buffer sing uga entuk manfaat kanggo paket liyane:

      • smartparens, paredit, lispyville, lsp ...: Iki paket sing nyedhiyakake api sing paling apik kanggo ngetrapake manipulasi teks adhedhasar obyek teks, kaya slurping, nanging alon-alon (khusus smartparens). Rebasing api ing ndhuwur wit katon apik banget.

      • Polymode: minangka mode suntingan kanggo campuran macem-macem mode utama ing buffer sing padha. Iki ditrapake kanggo nyuda wilayah kanthi ekspresi biasa, nggunakake wit ing kene bakal nyepetake.

      • Orgmode: sawise grammar kanggo sitter wit wis ditulis, orgmode bisa entuk kacepetan cepet amarga akeh logika lan eksekusi mode org adhedhasar konten buffer sing kudu diuraikan kanthi rutin (properti, blok kode, judhul, output)

    • Tambah hiperbola menyang orgmode: paket iki ngetrapake hyperlink global, dene hyperlink orgmode mung bisa digunakake ing buffer orgmode. Uga orgmode sing adhedhasar mode outline, bisa entuk manfaat saka mode kline sing diwenehake dening hyperbole, sing ngetrapake node non colliding.

    • Circhat: tembung mashup antarane circe.el lan weechat.el . Circe ngetrapake UI sing apik banget kanggo buffer obrolan, dene weechat.el nyedhiyakake dhukungan kanggo antarmuka protokol relay weechat , nanging duwe UI sing kurang apik lan apik. Nggunakake weechat kanggo nangani komunikasi obrolan bakal ngilangi emacs supaya ora bisa ngatasi kabeh sambungan IRC lan ngawasi kabeh buffer, sing bisa nyuda emacs cukup akeh. Kajaba iku weechat duwe dhukungan plugin kanggo matrik, mula sampeyan uga entuk dhukungan matrik langsung ing emak.

    • Tramp: tramp minangka paket emacs sing nangani koneksi jarak jauh, nanging alon banget. Iki ora mesthi kabeh kesalahan tramp, amarga paket liyane ora nate mriksa manawa sesi saiki lokal utawa remot, lan ngetrapake kaya operasi sistem file ing titik pungkasan. Nanging tramp uga ora duwe dhukungan async, yaiku kahanan sing sedhih banget amarga ana prekara sing biasane nggawe prentah kanggo proses eksternal sing nangani protokol sing dijaluk (ssh, sftp, docker ...). Pengalaman VSCode karo host host luwih apik, lan emacs kudu nambah game :)
