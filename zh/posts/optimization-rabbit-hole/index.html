python中的优化兔子洞

加快 freqtrade 加密交易机器人的回溯测试。

频率贸易 是 python 中的加密货币交易机器人。我曾经玩过一段时间的叉子，修复了一些错误和一些我不喜欢的行为。在这里列出错误会很无聊，所以我会谈谈我认为有趣的事情。

freqtrade (FQT) 的工作原理

FQT 不是基于事件的，而是基于循环的。这是什么意思？我们可以谈论执行者（实时操作）或回溯测试者。在这种情况下，它们都是基于循环的。

在这里绕道而行。我喜欢什么？循环还是事件？

  • 与事件相比，循环通常更简单，更容易推理。 Events OTOH 可以更好地模拟现实世界。与事件相比，循环也更容易并行化，并且通常执行得更快。

  • 基于循环的回测可以与实时执行器并排运行。这意味着后向评估成为您交易策略的一部分。运行基于事件的回测系统 居住 对我来说似乎一团糟。那是因为事件需要时间概念，但在实时交易中你只有现在，所以无论如何你会模拟什么以及如何模拟？

...返回到 FQT，在实时模式下，它循环处理订单。每 X 秒它查询更新交换。它跟踪：

  • 持有所有资产的所有最新价格的配对列表（对于支持它的交易所）

  • 新鲜的 OHLCV 数据

  • 未结订单状态

卢氏

基于循环的实时执行有一个明显的问题。它是同步的。如果您处理许多对，执行的响应速度会降低，迭代次数也会减少，从而滞后于价格行为的展开。因此，尽管我更喜欢回测循环，但我更喜欢实时执行事件。

不足之处？如果您为一件事使用事件而为另一件事使用循环，则策略的模型或参数更容易发散而不符合现实。

我对此的辩护理由是，不断运行自我评估（回测）和实时执行可以避免这个问题，因为它们会 最终 对齐。

FQT 有很多配置参数，很多时候机器人和策略之间的区别会变成灰色，因为策略无法控制机器人如何执行订单。该机器人处理“roi”，这是一个止盈价格限制和“追踪止损”的网格，但在不利方面相同。该策略仅提供参数，剩下的由机器人完成。我不喜欢这个。

买入和卖出信号的回调。交易将根据回调返回的信号发生，回调将计算数据帧上的信号。问题是信号是静态的；您可以动态计算一个 1 行数据框，并将其提供给机器人，但它不可回测。如果您不太重视对每个人进行回溯测试，这很好。

拼布

我在 FQT 中修改的内容试图让它表现得符合我的喜好。

  • 更多配对列表过滤器 .通过减少每次迭代中要处理的对数，您可以以牺牲一些反应性为代价来减少负载。您仍然会处理所有存在未平仓交易的资产，以及数量有限的新资产。所以我建造了

    • 一种 混洗过滤器 只是随机挑选对。

    • 一个循环过滤器，一点一点地迭代。

    • 静态列表，从存储中加载

  • 数据格式：我添加了对 hdf 的支持， 镶木地板 和 扎尔 .其中，我发现自己使用的是 zarr，这要归功于其内置的快速数据解/压缩功能。比镶木地板更简单，更符合人体工程学。

还有很多其他的调整，比如我最终放弃的并行信号评估，因为我更多地关注构建快速信号，以及绘图和配置选项的调整，但它们并不值得注意。

追求 矢量化 回测

我尝试了很多版本的回测来加快速度。这一切，一点也不值得。使用 numpy 数组计算买卖操作之间的逻辑。这就像用你的大脑玩俄罗斯方块。我的意思是积木是由你的灰质构成的，你试着把它们拼在一起。在这里我们列出它们

  • 分块回测 .在这一个中，我们将尝试仅使用 numpy 数组来关闭交易。在买入/卖出、信号、止损、尾随止损、投资回报率网格计算、时间加权投资回报率计算的正确执行流程之间游刃有余。都是 numpy 数组，真是一团糟。

  • 基于VBT 这个版本试图利用 矢量BT基于 numba 的 python 库来执行回测。令人惊讶的是，它并没有比 numpy 版本快多少。这是因为必须将大量类型转换为 VBT 兼容类型，因此从执行中获得的任何速度都会在那里丢失。

  • 遍历范围 这是我自己第一次尝试使用 numba。因为 numba 可以本地调用 numpy 函数，所以我调整了基于 numpy 的回测框架以在 numba 下工作。收益很小。将所有内容放入 numpy 数组的成本是瓶颈。之后取得的收益很小，因为在 numpy 转换之后，numpy 的执行已经很快了。

  • 在蜡烛上循环 这是理所当然的最后一个。这是表现最好的实施。那是因为它是对蜡烛图的“简单”迭代，所有这些都在 numba jitted 函数中执行。这个概念很简单，但是在处理 numba 错误时实现所有 roi 和止损逻辑并不好。它最终工作得很好，但在我不得不爬上一座 numba 不兼容的山之后。

我们还为回测添加了一些额外的功能，比如 点差和流动性 计算，以及前面提到的时间加权 roi。

追求 平行线 优化

为什么我想要一个快速回测器？因为我想运行其中的许多，以便我可以找到最好的参数配置中的最好的......当然，忽略任何过度拟合，过度参数化，缺乏关注等的概念......让我们列出我的工作：

  • 将工作分派给其他工人（进程） 尽可能快.第一个改进是改变工作分配的方式。从分派 N 个工作并等待所有工作完成，我改变了流程，以便在之前的工作完成后立即提供新工作。然而，这种方法存在问题。优化是多次连续的。如果不完成先前的计算，您将无法知道要观察参数空间的哪些点。至少对于贝叶斯优化来说是这样。这意味着如果我们连续对新试验进行采样，我们不会在优化之间改进搜索，但搜索会更加稳健。

    我们可以使用另一个不基于贝叶斯优化的优化器，比如 这个 .或者运行 N 个独立的优化器，这样每个进程/优化器都会使用自己的历史查询新作业，或者选择与所有其他进程共享的历史（尽管这会使所有优化器在一段时间后收敛，这本来是一个不错的选择财产，或不是，视情况而定）。

    在所有进程之间共享观察结果是一场彻头彻尾的灾难。对大量浮点数进行酸洗和反洗是非常缓慢的。随着搜索的进行，观察的数量越来越多，整个过程变得越来越慢，以至于序列化比观察评估花费更多的时间。使用管理器进程、普通文件或内存映射文件都很慢。最好的 物有所值 因此，方法被证明是运行单独的优化器并且从不在它们之间共享观察结果，将所有内容保存在内存中。

  • 跟踪优化状态。想象一下运行几个小时的优化，然后出现崩溃，你失去了一切。当然我们想避免这种情况，所以我们添加了逻辑 定期保存 .这是添加 zarr 存储支持的主要驱动力，用 zarr 保存状态大大改善了事情。

  • 执行优化的可视化。 FQT 已经具有良好的运行优化可视化效果。我改进了它。我必须创建一个 目的 代表每一个观察。然后我可以运行额外的后期处理，比如 正常化 和 过滤

  • 为了 交叉验证 我们必须拆分数据，并在不同范围内应用回溯测试。由于我们在优化状态跟踪方面的努力，我们可以使用交叉验证的输出作为另一个更精细的优化过程的种子……当然可以过度拟合。

  • 额外的优化逻辑。达到最大值 搅动 我们还必须考虑一些问题：

    • 有时参数配置可能会失败，优化器可能会卡住，因此我们必须继续跟踪失败的观察结果，并最终重新启动一个或所有优化器。

    • 因为我们运行了多个优化器，所以我们必须跟踪哪个优化器产生了哪个观察结果。

    • 多亏了我们的快速回测器和并行优化器，我们运行了很多观察结果，FQT 会在每个观察结果完成时打印出来。我们必须在打印之前对观察结果进行批处理，这样我们就可以打印一个"tableful" 它们而不是一次只有一个观察，因为 IO 操作是昂贵的。

    • 勘探/开发平衡。有些运行可能会陷入极小值，或者收敛速度很慢，因此我们添加了修改搜索空间的功能 在飞行中 .这将有助于使搜索速度正常化。

    • 一个更好的进度条。在多个进程上运行优化器需要同步来更新来自多个工作进程的进度条。还有一个图书馆只是为了 那 , 这让我们也可以添加 更多信息 到优化的运行状态。

    • 因为我们运行了多个优化器，所以我们还添加了支持，为不同的优化器提供不同的损失函数，这将使我们能够看到哪个损失函数表现最好。

  • 为了运行优化，我们实现了一个 命令行脚本 使用不同的配置参数运行优化，从以前的状态恢复，或者只是显示和过滤保存的试验。

  • 我们正在运行具有不同损失函数的独立优化器，那么，为什么不运行 不同的优化器 在 不同的损失函数 ？精神崩溃了。所以我们实施了一个 抽象 对于优化器，我们可以（通过一些管道）神奇地交换优化方法。这也需要修改优化 界面 这样“hyperopt”就可以理解优化器。然后我们重新实现 scikit-优化 作为我们的优化器抽象的一个实例。同时也插上 斧头, emukit 和 夏尔巴协作.

结论

我们启动了 FQT 回测 到 11 .但我们从未真正使用过它 生产中 :).

稍稍后 麻木在回测器中，额外的回调被添加到策略中，打破了回测和策略评估之间的分离，这意味着为了保持快速，你还必须在 numba 中编写你的策略！但是我受够了 python/numpy/numba 陷阱的不稳定组合，因为我不喜欢 FQT 的实时执行，我还是放弃了整个东西，为了更环保（或者我应该说 粉红！ ）牧场。

无论如何......优化是破解。
