我的编程语言

我选择的编程语言

为了保持清单简短，我尽量坚持 每个生态系统一种语言

茱莉亚

好吧，这不是一个生态系统，但因为它是我首选的语言，所以它必须排在第一位。我最喜欢它的原因是它是唯一的动态语言 真正符合人体工学 ，我喜欢对事物进行基准测试是多么容易，我喜欢 Makie，类型组合的方式以及调度系统如何与参数类型无缝集成。

茱莉亚 为了速度和易用性而牺牲一些内存和编译时间，并且很少看到小于〜500M ram的进程。因此，如果你讨厌 electron 或 java，你可能也会对 julia 有疑问，尽管有一些解决方案可以为你提供小型 julia 二进制文件，但它们都是实验性的，而且无论如何，julia 是一种动态的第一语言，所以我认为主要目标用例将是总是从 repl 运行东西，编译的二进制文件将永远是第二公民，但时间会证明一切。

最近出现了一种叫做 mojo 的语言，有人开始质疑它是不是对 julia 的“威胁”。如果您已经使用 julia 一段时间，然后浏览了 mojo 文档（就像我一样），您可能已经意识到 mojo 是一种静态语言 用螺栓固定 在动态 python 之上，所以很难将它称为与 python “统一”的语言，因此你正在比较苹果和橘子......

如果您想将 julia 与其他语言进行比较，还有其他一些基于“JAOT”的语言：

  • numba：与 julia llvm jit 相同，基于 python、numpy 和 numba 特定习语的混合，你不能真正基于它构建大型项目，只对加速小功能有用

  • gccemacs：本机编译模式下的 emacs 与 julia 的功能相同，但形式要弱得多，因为它不能真正从 lisp 中推断出非常具体的类型，绝对不能 emacs 口齿不清 ，并且基于 gcc jit。

尼姆

我们可以称之为“C”生态系统，我第二喜欢的语言是 尼姆 ，但我很少使用它，因为它的工具很垃圾。所以请有人在 nim 工具开发中投入几百万，谢谢！这是我最喜欢的，因为正如我之前提到的，julia 很难发布，而且您绝对不想发布千兆字节大小的二进制文件。相反，Nim 可以编译非常小的二进制文件并使它们成为静态的。

唯一的痛点（从语言的角度来看）是线程和异步之间的交互。在异步 couritine 和线程之间传递东西可能充满陷阱，因此它缺少某种系统可以更轻松地在线程和异步运行时之间进行兼顾。此外，使用异步运行时会使二进制文件变得非常大，因为异步闭包重写或类似的东西，底线是二进制文件有大量的代码生成函数，并且由于函数命名方案，调试它们也很困难。

除了总体上易于发布之外，它还具有强大的 ffi 支持、点调用语法、比 lisp 更好的宏/模板系统（如果你忽略缺少 sexprs）、julia 中的宏还可以，但 nim 可以“调度”宏和类型之上的模板，允许您在“污垢和卫生”之间进行选择，并且语言服务器能够在函数、宏和模板之间提供很多智能，即使是 Rust 也无法实现（如果它不会每 2 秒崩溃一次的话）慢慢来……叹气）。

Nim 也有 nimscript，我希望可以围绕它构建一个完整的交互式 repl，如果热代码重新加载再次被拾起，那就是......

锈

Rust 是我使用的，因为 nim 工具是垃圾，如果 nim 工具不是垃圾，我会使用 nim。我个人并不认为 Rust 内存安全比 nim 有这样的优势。 Nim 没有“fearless concurrency”，与 rust 相比，在处理线程时 nim 很容易出现段错误。然而，rust 使用生命周期来实现线​​程安全，而 nim 在语言级别上没有任何这样的东西。生命周期可能是最令人困惑和难以阅读的 Rust 语言特性，所以可能有一个原因......

一般来说，您可以说在某一点之后，业务错误变得比内存错误更常见，并且您不希望使语言过于复杂以实现内存安全收益递减，同时破坏可读性并增加语言本身的复杂性。因为量化语言使业务错误更常见的程度几乎是不可能量化的，所以这一点总是没有实际意义。

其余的部分

这些只是我列出的“替代”语言列表中的语言 可能 每个生态系统选择一次

  • Beam：在 gleam 和 elixir 之间做出艰难的选择，可能最终会选择 gleam 或尝试两者并选择一种，当然它们是根本不同的语言，但我确实提到过每个生态系统只会选择一种。

以下是我会使用这些生态系统，但还没有真正尝试过的：

  • JVM/Graal：Clojure。紧随其后的可能是过去的 Scala，但 Clojure 实际上是唯一的“工业”语言，所以必须选择那个。

  • Node/Deno/Bun：我会选择 typescript，因为它是最常见的 compile-to-js 语言。将来我可能会考虑 ReScript，如果它的生态系统会变得足够大的话。

  • CLR：没有？我会说 F#，但我更愿意说没有 CLR。 clr 似乎也有 clojure 和 swift 的实现，所以就是这样，但我的印象一直是使用这些“端口”中的任何一个都会遇到一堆不兼容问题，无论如何我不认为自己曾经接近过 clr个人选择。

  • Lisp：Lisp 是一个生态系统吗？嗯...不是真的，尽管有人可能会说这都是 sexprs，并且将事物从一个方案移植到另一个方案就像您将要获得的搜索和替换查询一样接近。如果 emacs lisp 有一个快速的方言，可以利用 gccemacs 吐出快速代码，我会强烈考虑它而不是普通的 lisp。如果我不得不选择一个方案，那将是毫无疑问的。我是否认为自己在用 commonlisp 或 racket 编写实际的软件？不太可能，因为如果我需要 lisp，我会选择 clojure。

除了这些“生态系统”语言选择之外，还有其他一些值得一提。

  • Raku：我过去做过一些适度的 PHPing 和零 perl，但 raku 似乎真的很有趣，如果不是因为该语言无与伦比的代码通用性的话。

  • 红色：全栈承诺很吸引人，rebol 的传统非常独特，值得在其中编写一个大小适中的应用程序来尝试一下

  • Dart：如果我偶然发现了一些 UI 开发人员，我可能会尝试一下，但我更愿意坚持使用我的顶级语言之一，即使是 UI 开发人员，那么问题就变成了， 多少 由于多平台支持，dart（或 haxe！）会提高生产力吗？

  • Verse：来自史诗游戏的语言，独一无二，因为实际上是唯一现代的序言式语言。史诗之所以如此建造似乎很明显，因为每个人都讨厌 UE 蓝图。但是，如果该语言与 UE 保持紧密耦合，我认为自己从未尝试过。

还有许多其他流行的语言，比如一些更实用的语言，如 ocaml、haskell 或其他更简单的语言，如 zig 或 vlang，我没有提到，因为我认为它们是 太死板了 超越他们的范式（不变性/简单性），总的来说，我想我更喜欢更“全面”的语言。

这个列表最终并没有那么短，我猜很抱歉。
