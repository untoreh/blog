เชลล์สคริปต์

ความคิดเกี่ยวกับเชลล์สคริปต์

ทุบตี/ zsh เชลล์สคริปต์ทำให้งานเสร็จ มันยังค่อนข้างน่าเกลียด บางทีก็ตั้งใจ ยิ่งคุณอยากดูน้อยเท่าไหร่ สคริปต์ก็จะยิ่งสั้นลงเท่านั้น ฉันไม่ได้เป็นแฟนของการใช้ bash/zsh เป็นเชลล์แบบโต้ตอบ จำนวนปลั๊กอินที่จำเป็นสำหรับการชดเชยประสบการณ์ที่ดีนั้นน่ากลัว และปลั๊กอินเพิ่มเติมทุกตัวเป็นหลุมพรางสำหรับเชลล์ที่ช้าลง ไม่น้อยกว่าปริมาณงานที่ทำโดย psprint ด้วย zinit & ผองเพื่อน น่าประทับใจมาก (และเป็นทางเลือกเดียวที่สมเหตุสมผลสำหรับตัวจัดการปลั๊กอิน หากคุณต้องการใช้ zsh เป็นเชลล์แบบโต้ตอบ) อย่างไรก็ตาม มันไม่ได้แก้ไขความจริงที่ว่าปลั๊กอินที่โหลดทั้งหมดเหล่านั้นเป็น ไม่แชร์ หน่วยความจำระหว่าง (อาจ) กระบวนการเชลล์จำนวนมาก

เปลือกปลาเป็นเชลล์ที่ฉันชอบในขณะที่เขียน จำนวนปลั๊กอินที่ติดตั้งนั้นน้อยมาก เพราะทำสิ่งที่คุณต้องการได้มากที่สุดจากเชลล์ ออกจากกล่อง . สคริปต์ค่อนข้างช้ากว่า zsh แต่ไม่มีใครสนใจเกี่ยวกับความเร็วของสคริปต์ ดังนั้นจึงเป็นเรื่องปกติ ไวยากรณ์เป็นสิ่งที่ดีสำหรับมรดก ML (เช่น Julia) ที่มีข้อความสุดท้าย แทนที่จะเป็นวงเล็บปีกกา การเขียนสคริปต์นั้นละเอียดกว่า bash/zsh แต่ก็สามารถอ่านได้ง่ายกว่ามาก สิ่งต่างๆ เช่น การแทนที่กระบวนการ การสอดแทรกสตริง และการวางไพพ์จบลงด้วยรายละเอียดที่ละเอียดขึ้นอีกเล็กน้อย มันมี ทั่วโลก ตัวแปรที่แบ่งใช้ระหว่างกระบวนการที่ทำงานอยู่ทั้งหมดและ สากล ตัวแปรที่แชร์ข้ามเซสชัน ซึ่งทำให้สิ่งต่างๆ ง่ายขึ้นโดยไม่ต้องเขียนทุกอย่างลงไปและซอร์สสคริปต์ที่ลืมไม่ลง

ภูติ เป็นเชลล์ที่รองรับหลายๆ อย่าง เช่น daemon หรือ web mode ที่อนุญาตให้ส่งคำสั่งจากระยะไกล...? ฉันคิดว่ามันมากเกินไปและฉันเริ่มเรียกมันว่าป่อง IMHO เปลือกควรมุ่งมั่นเพื่อความเรียบง่ายและเป็นตัวของตัวเอง ฉันคิดว่าเชลล์ควรได้รับการออกแบบเพื่อจัดการกับสถานการณ์ที่เป็นกระบวนการที่เกิดขึ้นในจำนวนครั้งที่ชัดเจน ซึ่งทำงานอย่างสม่ำเสมอตั้งแต่ต้นจนจบและมีอยู่ในตัวเองเพื่อให้สามารถจัดการกับข้อขัดข้องได้อย่างสวยงาม ดังนั้นแม้จะมีฟีเจอร์อื่นๆ ทั้งหมด (เช่น ข้อมูลที่มีโครงสร้าง)...daemons ก็ไม่เหมาะกับฉัน

ไอออนเชลล์ เป็น เกือบ posix เชลล์จากรีดอกซ์ ซึ่งทำงานบนยูนิกซ์ด้วย โดยมีเป้าหมายเป็น อีกด้วยทำงานบน windows แม้ว่าจะใช้งานไม่ได้ ATTOW เป้าหมายของมันคือเพียงแค่เป็นเปลือกที่ดีกว่า ดังนั้นมันจึงนำข้อดีจากโพซิกซ์มาเพิ่มเข้าไป ดูเหมือนว่าจะเน้นที่การจัดหาภาษาสคริปต์ที่ดี แต่ไม่ใช่ประสบการณ์เชิงโต้ตอบที่ดีนัก เปลือกปลาทำให้ฉันเสีย และฉันจะไม่กลับไปหาเปลือกโดยที่ไม่มี ชั้นหนึ่ง การโต้ตอบ

เปลือกน้ำมัน มีเป้าหมายเพื่อแทนที่ bash ซึ่งน่ายกย่อง แต่มีเป้าหมายเฉพาะระบบยูนิกซ์ที่ไม่มี Windows อยู่ในสายตา ได้มุ่งเน้นไปที่การแก้ไขเลเยอร์ความเข้ากันได้ของ bash มาจนถึงปัจจุบัน โดยมีแผนในอนาคตที่จะมุ่งเน้นไปที่ส่วนการโต้ตอบ เมื่อพิจารณาจากข้อเท็จจริงที่ว่าส่วนโต้ตอบของเชลล์ยังไม่ได้ถูกเขียน ทำให้ฉันสงสัยเล็กน้อยเกี่ยวกับอนาคตของมัน และแม้ว่าฉันหวังว่ามันจะไม่ไม่เป็นเช่นนั้น ฉันทำได้สำเร็จก่อนที่มันจะบรรลุเป้าหมายที่ตั้งใจไว้ การขาดการรองรับ windows นั้นไม่เหมาะเช่นกัน

ผื่น เชลล์ที่รองรับข้อมูลที่มีโครงสร้างซึ่งใช้งานได้กับ แร็กเกต เมื่อฉันค้นหาแร็กเกตมากขึ้นฉันจะลองแม้ว่าประสบการณ์ที่ผ่านมาของฉันกับ [ emacs เชลล์ ] ไม่สวยจริงๆ เนื่องจาก เปลือกไข่ ใช้งานได้ภายใน emacs เท่านั้นและใช้งานไม่ได้อย่าง a เปลือกระบบ ...เว้นแต่ระบบของคุณจะเป็น emacs (แม้ว่า emacs จะเป็นตัวแก้ไขหลักของฉันและฉันก็ยังเขียน) บาง ไม่ใช่ระบบผ่าตัดของฉัน) (ที่นี่ฉันควรพูดถึง Julia REPL ซึ่งเป็นตัวแทนที่ดีที่สุดสำหรับภาษาการเขียนโปรแกรมใด ๆ )

Powershell Powershell เชลล์หลักของ Windows ทำงานบน linux และ macs ด้วย ต้องขอบคุณข้ามแพลตฟอร์ม สุทธิ . แต่มันละเอียดเกินไป ช้า และอาจจะ (ไม่ได้รับการยืนยัน) เกลื่อนไปด้วย windowsisms ; มันเป็นเรื่องใหญ่สำหรับฉัน

นูเชลล์...แน่นอนว่าเราสำรองสิ่งที่ดีที่สุดไว้เป็นครั้งสุดท้าย นี่อาจเป็นเชลล์ต่อไปของฉัน ความสนใจระดับเฟิร์สคลาสสำหรับการโต้ตอบ การพิจารณาแต่ละเชลล์ในตัวและข้ามแพลตฟอร์มอย่างรอบคอบ และข้อความที่มีโครงสร้าง

กระสุนที่ไม่ได้อยู่ในรายการนี้ ATTOW ถือว่าไม่น่าสนใจพอ
