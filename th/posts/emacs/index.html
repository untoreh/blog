Emacs

ทำไม emacs และสิ่งที่อยากได้ emac ส่วนตัวของฉัน

emac คืออะไร?

NS เครื่องเสมือน หรือล่ามสำหรับ emacs ภาษาโปรแกรม lisp ... ชนิดของ ภาษาการเขียนโปรแกรม lisp ของ emacs (elisp) เป็นวัตถุประสงค์ทั่วไป แต่มีการสนับสนุนชั้นหนึ่งสำหรับโปรแกรมแก้ไขข้อความจริงที่เรียกใช้ ประเภทดั้งเดิมนั้นเน้นไปที่การแก้ไขข้อความ อย่างไรก็ตาม คุณสามารถเขียนอะไรก็ได้ที่คุณต้องการด้วยความน่ารัก ประสิทธิภาพที่ดี (ล่าสุดต้องขอบคุณ jit คอมไพล์ ) เพราะเป็นทายาทของ MACLISP และพี่น้องของ เสียงกระเพื่อมทั่วไป . อย่างไรก็ตาม lisp ของ emacs ไม่มีมาตรฐาน และข้อกำหนดนี้เทียบเท่ากับการใช้งานที่ได้รับความนิยมสูงสุด นั่นคือ GNU Emacs

ทำไมต้องอีแมค?

ด้วยวลี "Emacs เป็นระบบปฏิบัติการ" ผู้คนหมายความว่าคุณสามารถใช้เพื่อทำทุกอย่างที่คุณต้องการใช้คอมพิวเตอร์ มันขาดการเป็นระบบปฏิบัติการตามตัวอักษรเนื่องจากเคอร์เนล (โดยปกติคือ linux) ยังคงรับผิดชอบฮาร์ดแวร์[1] อย่างไรก็ตาม สามารถเรียกได้ว่าเป็นสภาพแวดล้อมเดสก์ท็อปได้อย่างง่ายดาย เนื่องจากมีสภาพแวดล้อมในการทำงานด้วย ช่วยให้คุณสามารถเขียนโค้ดแบบโต้ตอบได้อย่างรวดเร็วเพื่อเชื่อมต่อกับแอปพลิเคชันใดๆ ก็ตาม มันอาจจะสกปรกพอๆ กับเชลล์สคริปต์ หรือคิดมาอย่างดีเพื่อให้ API ที่เสถียร อันที่จริงแล้ว ฟังก์ชัน emacs จำนวนมากมีให้ในแพ็คเกจ[2] โปรแกรมแก้ไขอื่นๆ สามารถขยายได้เหมือนกับ emacs หรือไม่ ไม่ทำไม? เนื่องจากเอดิเตอร์รายอื่นใช้โมเดลผู้ใช้ที่แตกต่างกัน ผู้ใช้จึงต้องถูกจำกัดเนื่องจากถือว่าเป็น "แขก" ของสภาพแวดล้อมที่ทำงานอยู่ซึ่งเสนอ "บริการ" ให้กับผู้ใช้ นี่ไม่ใช่กรณีของ emacs ที่สภาพแวดล้อมและผู้ใช้กลายเป็นหนึ่งเดียวกัน ใน emacs คุณสามารถเข้าถึงทุกอย่าง และสามารถปรับเปลี่ยนได้เกือบทุกอย่างด้วยวิธีเดียวกัน สิ่งเหล่านี้ถูกสร้างขึ้น หรือที่เรียกว่า elisp

มันคุ้มค่าหรือไม่?

ผลประโยชน์ที่สะสมมา คือ การลงทุน และเช่นเดียวกับการลงทุนอื่นๆ คุณควรคาดหวังกำไรตามสัดส่วนกับจำนวนเงินที่คุณลงทุนไป ดังนั้น คุณต้องใช้เวลาในการเรียนรู้และฝึกฝนวิธีใช้งาน

GNU emacs สิ่งที่อยากได้

Emacs เป็นระบบปฏิบัติการที่ดีมาก แต่เป็นโปรแกรมแก้ไขข้อความที่ไม่ดี

...หรืออะไรทำนองนั้น การนำ GNU Emacs ไปใช้ เช่นเดียวกับ emacs มีรากฐานมาจากยุค 80 และส่วนต่างๆ ของแกนกลางแสดงอายุ...

  • ไดโคโทมีระหว่างอินเทอร์เฟซแบบกราฟิกและอินเทอร์เฟซเทอร์มินัล เนื่องจาก emacs นั้นเก่าแล้วจึงเริ่มต้นเป็นแอปพลิเคชันเพื่อเรียกใช้ภายในเทอร์มินัล แต่ด้วยการถือกำเนิดของอินเทอร์เฟซแบบกราฟิกในยุคนั้น มันจึงเริ่มรวบรวมฟังก์ชันการทำงานที่มุ่งไปที่ ตามหน้าต่าง ปฏิสัมพันธ์และประมาณปี พ.ศ. 2543 การสนับสนุนสำหรับ X11 มีการแนะนำโปรโตคอล เมื่อเวลาผ่านไป ประโยชน์ของรุ่นเทอร์มินัลอาจลดลง และด้วยเหตุนี้ ความสนใจในการบำรุงรักษามัน เนื่องจากการมี emacs ทำงานแตกต่างกันขึ้นอยู่กับกราฟิก/เทอร์มินัลค่อนข้างเป็นภาระและชิปที่มีความซับซ้อน ฉันต้องการเห็นนามธรรมของ emacs GUI เหนือบางอย่างเช่น sixelโปรโตคอลเพื่อให้อินเทอร์เฟซแบบกราฟิกและเทอร์มินัลสามารถบรรลุการแบ่งปันรหัสในระดับที่ลึกยิ่งขึ้น สิ่งนี้จะต้องใช้ emacs เพื่อกลับสมมติฐาน "เทอร์มินัลแรก, กราฟิกวินาที" (ซึ่งในขณะที่เขียนฉันคิดว่ายังคงมีอยู่)[3]

  • รหัส Async ใน elisp นั้นไม่ใช่เรื่องง่ายหรือเป็นไปได้ในบางกรณี Emacs รองรับเธรดที่ ผลผลิต นั่นคือสามารถหยุดชั่วคราวและคืนการควบคุมให้กับตัวกำหนดตารางเวลาหลักซึ่งเป็นการดำเนินการของ มัลติทาสกิ้งแบบร่วมมือกัน . emacs ต้องการการสนับสนุนที่ทันสมัยสำหรับ async primitives หรือไม่ ความจริงที่ว่า emacs ส่วนใหญ่ถูกจำกัดไว้ที่แกนเดียว หมายความว่าโค้ดนั้นได้รับการปรับให้เหมาะสมที่สุด และ emacs เองก็สามารถทำงานได้ค่อนข้างเร็วบนอุปกรณ์ขนาดเล็กเช่นกัน ฉันคิดว่า emacs ซึ่งเป็นอินเทอร์เฟซแบบโต้ตอบต้องรองรับเฉพาะ an ส่วนต่อประสานกราฟิกแบบอะซิงโครนัส . ในกรณีนี้ ขอบเขตและบริบทสำหรับโค้ดอะซิงโครนัสจะถูกจำกัดไว้เฉพาะ UI นี่คือการออกแบบที่ใช้ในบรรณาธิการที่ใช้อิเล็กตรอนอื่น ๆ เช่น VSCode โดยที่ UI ทำงานในกระบวนการที่แยกจากกัน ฉันต้องการโดยเฉพาะอย่างยิ่งสำหรับ ประสบการณ์การพิมพ์ เพื่อไม่ให้ถูกขัดจังหวะ และฉันยินดีแลกกับการเน้นไวยากรณ์ที่ล่าช้า การเสร็จสิ้น หรือเครื่องมือเสริมอื่นๆ สำหรับ ซอฟท์เรียลไทม์ เวลาแฝงของอินพุต

  • เนื่องจากใน emacs คุณควรเข้าถึง ทุกอย่าง วิธีที่ emacs พูดใน X11 นั้นค่อนข้างโลภมาก เนื่องจาก emacs เกือบจะนำเซิร์ฟเวอร์ X11 แบบสมบูรณ์มาใช้ใหม่ภายในตัวมันเอง สิ่งนี้ทำให้เกิดปัญหาในอดีต[4] เนื่องจากส่วนใหญ่ใช้ emacs เป็นหน้าต่างภายใน X11 อื่นและไม่ใช่ เซิร์ฟเวอร์ X11 เอง . เซิร์ฟเวอร์แสดงผล ซึ่งจัดการเซสชันแบบกราฟิก สภาพแวดล้อมการแสดงผลที่แตกต่างกันจะมีลักษณะอย่างไร ฉันจินตนาการถึงบางสิ่งที่เริ่มต้นจากระดับที่ต่ำกว่า และดูเหมือนเอ็นจิ้นเกมมากกว่า อาจจะขึ้นอยู่กับ SDL . ด้านบนของเอาต์พุตวิดีโอโดยตรงพื้นฐาน ที่สามารถเข้าถึงได้จาก elisp (ที่คอมไพล์โดยกำเนิด) หรือ API C (หรือสนิม) ที่เสถียร ระบบหน้าต่าง emacs จะถูกนำมาใช้ที่ด้านบนของแบ็กเอนด์ "pixel cruncher" และไม่ต้องพึ่งพาชุดเครื่องมือกราฟิกภายนอกอีกต่อไป เช่น lucid หรือ GTK และบรรลุการไม่เชื่อเรื่องพระเจ้าของแพลตฟอร์มในระดับที่สูงขึ้น นอกจากนี้ยังหมายถึงประสิทธิภาพที่เชื่อถือได้มากขึ้นในทุกระบบ เนื่องจากเมื่อจอแสดงผลได้รับการจัดการโดยชุดเครื่องมือกราฟิกภายนอก ประสิทธิภาพของชุดเครื่องมือกราฟิกภายนอกอาจคาดเดาไม่ได้ และตัวอย่างเช่น วางไข่กรอบใหม่ ลงเอยบน windows ช้ากว่าบน linux; เมื่อมีการใช้งานฟังก์ชันเหล่านี้บ่อยครั้ง เช่น การแสดงตัวเลือกที่สมบูรณ์ ความคลาดเคลื่อนจะปรากฏชัด[5]

  • การมีแพลตฟอร์มการแสดงผลที่ทันสมัยกว่าจะไม่เพียงแต่ให้ประสิทธิภาพที่ดีขึ้นเท่านั้น แต่ยังสามารถเข้าถึงเครื่องมือภาพขั้นสูง เช่น การเปลี่ยนภาพ เงา การเบลอ และระบบที่ดีขึ้นสำหรับเลเยอร์ เนื่องจากปัจจุบัน Emacs ใช้งานพวกมันด้วย ซ้อนทับ ซึ่งขนาดไม่ดี โดยทั่วไป คงจะดีถ้ามีพลังเต็มที่ของ CSS3 และอื่น ๆ พร้อมใช้และปรับปรุงอินเทอร์เฟซ emacs แบบสปาร์ตันในปัจจุบัน[6]

  • ทำงานกับข้อกำหนดบางอย่างสำหรับ elisp ผู้ดูแล emacs ในปัจจุบันให้ความสำคัญกับความสามารถในการเรียกใช้ emacs บนแพลตฟอร์มต่างๆ (สถาปัตยกรรมซีพียู) หากแกนหลักถูกเขียนใหม่เป็น elisp เอง[7]สิ่งเดียวที่เหลืออยู่ภายในจะกลายเป็นล่าม elisp ด้วยข้อตกลงเกี่ยวกับข้อกำหนดเฉพาะของ elisp บางรูปแบบ การย้าย emac ทั้งหมดไปยังรันไทม์อื่นจะเป็นไปได้ เนื่องจากปัญหาเทียบเท่ากับ "เพียง" การนำล่าม elisp ไปใช้

  • แพลตฟอร์มการระดมทุนของ emacs? ในเรื่องนี้ โพสต์ Reddit มีการกล่าวถึงว่ามันยากแค่ไหนที่จะ "ให้ทุน" กับผู้คนในการทำงานด้านต่างๆ และจะต้องเป็นนักพัฒนาที่สนใจทำงานเกี่ยวกับคุณลักษณะเฉพาะที่ต้องขอเงินทุน อย่างไรก็ตาม สิ่งนี้ไม่ได้พิจารณาถึงข้อเท็จจริงที่ว่ามีคนจำนวนไม่มากที่เชี่ยวชาญด้าน emacs ภายใน การรอให้นักพัฒนาได้รับความรู้เพื่อใช้คุณสมบัติเฉพาะอย่างมีประสิทธิภาพอาจใช้เวลา 20 ปี (และกำลังเพิ่มขึ้น) อะไรจะเร่งขึ้นก็เพื่อ แรก จ่ายเงินให้ผู้คนเรียนรู้ (และเขียนเอกสารประกอบ!) เกี่ยวกับ emacs ภายในเพื่อให้ผู้คนจำนวนมากขึ้นสามารถทำความคุ้นเคยกับมันและค่อยๆ ย้ายการพัฒนา emacs จาก มหาวิหารและตลาดสด รูปแบบการพัฒนาให้ครอบคลุมมากขึ้น

  • แพ็คเกจรายการที่ต้องการ:

    • ต้นไม้นั่ง ทุกอย่าง! การเน้นไวยากรณ์ของ Emacs ขึ้นอยู่กับนิพจน์ทั่วไป ซึ่งไม่ได้ช้า (โดยส่วนใหญ่) แต่อาจทำให้ประสิทธิภาพการทำงานบางอย่างแย่ลงจนหลีกเลี่ยงไม่ได้โดยสิ้นเชิง แต่คนดูแลต้นไม้ไม่เพียงแต่ช่วยในการเน้นไวยากรณ์เท่านั้น แต่ยังให้การแสดงลำดับชั้นของเนื้อหาบัฟเฟอร์ซึ่งเป็นประโยชน์ต่อแพ็คเกจอื่นๆ ด้วย:

      • smartparens, paredit, lispyville ฯลฯ...: นี่คือแพ็คเกจที่มี API ที่โดดเด่นที่สุดสำหรับการใช้การปรับแต่งข้อความตามวัตถุข้อความ เช่น slurping แต่ช้า (โดยเฉพาะ smartparens) การปรับฐาน API ใหม่บนต้นไม้ดูน่าดึงดูด

      • Polymode: เป็นโหมดรองสำหรับผสมโหมดหลักที่แตกต่างกันในบัฟเฟอร์เดียวกัน ใช้ขอบเขตที่แคบลงผ่านนิพจน์ทั่วไป การใช้ตัวสร้างต้นไม้ที่นี่จะช่วยเร่งความเร็วของสิ่งต่างๆ

      • Orgmode: หลังจากเขียนไวยากรณ์สำหรับผู้ดูแลต้นไม้แล้ว orgmode อาจได้รับการกระแทกความเร็วอย่างมากเนื่องจากตรรกะและการดำเนินการของโหมดองค์กรจำนวนมากขึ้นอยู่กับเนื้อหาบัฟเฟอร์ที่ต้องแยกวิเคราะห์เป็นประจำ (คุณสมบัติ บล็อกโค้ด ส่วนหัว เอาต์พุต)

    • เพิ่ม อติพจน์ ถึง orgmode: แพ็คเกจนี้ใช้ไฮเปอร์ลิงก์ทั่วโลก ในขณะที่ไฮเปอร์ลิงก์ orgmode ทำงานเฉพาะในบัฟเฟอร์ orgmode เท่านั้น นอกจากนี้ orgmode ซึ่งอิงตามโหมดเค้าร่าง อาจได้รับประโยชน์จากโหมด kline ที่จัดเตรียมโดยอติพจน์ ซึ่งใช้โหนดที่ไม่ชนกัน

    • Circhat: การผสมคำระหว่าง circe.el และ weechat.el . Circe ใช้ UI ที่ดีมากสำหรับบัฟเฟอร์การแชท ในขณะที่ weechat.el ให้การสนับสนุนสำหรับการเชื่อมต่อกับ โปรโตคอลรีเลย์ weechat แต่มี UI ที่แย่และบั๊กกี้ การใช้ weechat เพื่อจัดการการสื่อสารด้วยแชทช่วยลด emacs จากการต้องจัดการการเชื่อมต่อ IRC ทั้งหมด และตรวจสอบบัฟเฟอร์ทั้งหมด ซึ่งอาจทำให้ emacs ช้าลงได้ค่อนข้างมาก นอกจากนี้ weechat ยังมีการรองรับปลั๊กอินสำหรับเมทริกซ์ ดังนั้นคุณจะได้รับการรองรับเมทริกซ์โดยตรงภายใน emacs ด้วย

    • คนจรจัด: คนจรจัดเป็นแพ็คเกจ emacs ที่จัดการการเชื่อมต่อระยะไกล แต่ช้ามาก นี่ไม่ใช่ความผิดพลาดของ tramps ทั้งหมด เนื่องจากแพ็กเกจอื่นๆ ไม่เคยตรวจสอบว่าเซสชันปัจจุบันอยู่ในเครื่องหรือจากระยะไกล และใช้สิ่งต่างๆ เช่น การทำงานหนักของระบบไฟล์บนปลายทางระยะไกล แต่คนจรจัดยังขาดการรองรับแบบ async ซึ่งเป็นสถานการณ์ที่น่าเศร้าสำหรับบางสิ่งที่ส่วนใหญ่ปิดบังคำสั่งไปยังกระบวนการภายนอกซึ่งจัดการโปรโตคอลที่ร้องขอ (ssh, sftp, นักเทียบท่า...) ประสบการณ์ VSCode กับการลบโฮสต์นั้นดีกว่ามากและ emacs จำเป็นต้องปรับปรุงเกม :)
