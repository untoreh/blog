ทำให้ Windows ทนทาน

รายการเครื่องมือที่ทำให้ Windows รู้สึกเหมือนเป็น Linux

ตั้งแต่ฉัน เปลี่ยนเป็น windows 10 ด้วย WSL ฉันคิดว่าฉันจะทำรายการซอฟต์แวร์ที่ฉันต้องใช้เพื่อทำให้ windows เข้าใกล้เวิร์กโฟลว์ linux ปกติของฉันมากขึ้น ให้ชัดเจนว่า Windows UI ห่วย มันติดอยู่ใน90s และทุกอย่างควรจะคลิกและลากด้วยตัวชี้เมาส์ของคุณ ไม่รู้ว่าเป็น วัฒนธรรม สิ่ง (windows devs ต้องชอบเมาส์มาก) หรือสำหรับ กลุ่มเป้าหมาย (“คลิกที่สิ่งนั้น!” อธิบายง่าย) หรือเพราะ แนวปฏิบัติด้านความปลอดภัยที่ดีที่สุด (หากเราให้การควบคุมแก่ผู้ใช้มากเกินไป มัลแวร์จะโจมตีมัน) หรือเพียงแค่ Windows Shellเกิดจากกองหนี้เทคโนโลยีซึ่งทำให้ยากต่อการเพิ่มฟังก์ชันสำหรับผู้ใช้ขั้นสูง ความจริงก็คือสิ่งง่ายๆ ใน Linux ต้องการโซลูชันที่ซับซ้อนใน Windows

เดสก์ท็อปเสมือน

ใน linux ตัวจัดการ windows ทุกตัวมีเดสก์ท็อปเสมือนบางรูปแบบ windows ได้สร้างเดสก์ท็อปเสมือนขึ้นมาค่อนข้างเร็วด้วย W10 ประเด็นคือในขณะที่เขียนคุณสามารถหมุนไปทางซ้ายและขวาด้วยปุ่มลัดเท่านั้นและไม่มีปุ่มลัดให้ไปที่เดสก์ท็อป X มีสิ่งนี้ ห้องสมุด vd แต่ฉันไม่พบซอฟต์แวร์ที่ใช้งานได้ แทนที่จะเป็นเพียง AHK[1] สคริปต์ที่มีการนำไปใช้งานที่ไม่ดี เช่น "วนซ้ำจนกว่าเราจะอยู่ที่ VD ที่ถูกต้อง"

การขาดทางลัดที่เหมาะสมสำหรับเดสก์ท็อปเสมือนในตัวของ windows ทำให้ฉันต้องใช้ซอฟต์แวร์ของบุคคลที่สามซึ่งมีอยู่มากมาย ฉันเลือก dexpot เพราะมันทำทุกสิ่งที่ฉันต้องการและอื่น ๆ อีกมากมาย (ฉันจะมีความสุขกับเพียงgo to X และmove window to X ทางลัด) ปัญหาเดียวคือ dexpot ไม่สามารถผูกคีย์ที่ผูกไว้อยู่แล้ว ... ค่อนข้าง AHK สามารถแทนที่คีย์ที่ใช้โดยแอปพลิเคชันอื่นได้ ในขณะที่ dexpot ไม่ทำเช่นนั้น ฉันคิดว่าเป็นเพราะใช้ windows API อื่น (หรือ [dll]). วิธีแก้ปัญหาคือทำการแมปทางลัดใหม่ที่ฉันต้องการ (เช่นWin+1 ) พร้อมทางลัดฟรีที่สามารถตั้งค่าบน dexpot (เช่นWin+Shift+F1)

การแมปแป้นพิมพ์

Powertoys อนุญาตให้แมปคีย์และทางลัดเพื่อให้ฉันสามารถแมป CapsLock->LeftControl และ RightControl->LeftControl และทางลัดอย่างWin+hjkl ถึงลูกศร เลวมากWin+l เป็นทางลัดเริ่มต้นของ Windows สำหรับการล็อกหน้าจอและไม่สามารถทำการแมปใหม่ได้...รีจิสตรีคีย์จะแทนที่ทางลัดและแก้ไขปัญหานี้...จนกว่าคุณจะอัปเดตหน้าต่าง...ดังนั้นคุณต้องใช้การแก้ไขในการบู๊ตแต่ละครั้ง!

ตัวเปิด

ตัวเรียกใช้งานแอปที่จัดหาให้โดย powertoys ค่อนข้างมีคุณลักษณะ ไม่ใช้หน่วยความจำมากไปกว่า KRunner และรวดเร็วและตอบสนอง ข้อร้องเรียนเพียงอย่างเดียวคือบางครั้งมันก็สูญเสียโฟกัส แต่นี่น่าจะเป็นปัญหากับ Windows ที่ป่วย (เช่นเดียวกับในทางที่ผิด) ว่าควรเน้นหน้าต่างใดในช่วงเวลาหนึ่ง

ปูกระเบื้อง

ฉันเปลี่ยนจาก sway เป็น KDE บน Linux ดังนั้นฉันจึงไม่ใช่ผู้ชื่นชอบ windows แบบเรียงต่อกันมากนัก Powertoys มี FancyZones ซึ่งเป็นการปรับปรุงเล็กน้อยจากหน้าต่างพื้นฐาน Snaps ที่หน้าต่างนำเสนอ เพราะมันช่วยให้คุณ ช่องว่าง และ เลย์เอาต์ . แต่ยังขาดสิ่งสำคัญที่ทำให้การปูกระเบื้องมีประโยชน์คือ กฎ เพื่อนำไปใช้กับหน้าต่างที่ตรงกัน แต่สิ่งนี้จะต้องได้รับการสนับสนุนสำหรับเดสก์ท็อปเสมือน...และเราได้กล่าวถึงสถานะปัจจุบันของพวกเขาแล้ว ในการเปรียบเทียบ KWin ช่วยให้คุณสามารถจับคู่หน้าต่างที่มีคำจำกัดความที่ซับซ้อนมาก FancyZones แก้ปัญหากฎโดยคงตำแหน่งหน้าต่างไว้ตลอดเซสชันต่างๆ แต่ฉันยังไม่ได้ตรวจสอบว่าวิธีนี้ทำงานได้ดีเพียงใดกับเดสก์ท็อปเสมือนต่างๆ

การลากหน้าต่าง

สามารถลาก Windows จากแถบหัวเรื่องเท่านั้นและเพิ่มเติม คุณประโยชน์ จำเป็นต้องใช้เพื่อย้ายด้วยชุดค่าผสมทางลัด

การตรวจสอบระบบ

ด้วย KDE คุณจะมีวิดเจ็ตพลาสมาเพื่อแสดงข้อมูลระบบ แม้ว่าวิดเจ็ตพลาสมามักจะใช้หน่วยความจำมาก ฉันจึงไม่ได้ใช้มันมากขนาดนั้น สำหรับ Windows จะไม่มียูทิลิตี้ในตัวสำหรับสิ่งนี้ หลังจากลองใช้ยูทิลิตี้สองสามตัวฉันก็ตกลง การตรวจสอบการจราจรซึ่งให้เครือข่าย ซีพียู หน่วยความจำ และเพิ่งถูกเพิ่ม GPU และอุณหภูมิ ผลลัพธ์ที่ได้คือสี่เหลี่ยมที่มีรายละเอียดต่ำที่สวยงามซึ่งแสดงอยู่ในแถบงาน:

คะแนนโบนัสสำหรับ Windows คือระบบจดจำ UPS ของฉันและแสดงในพื้นที่ถาด ดังนั้นจึงไม่ต้องการการกำหนดค่าเพิ่มเติม ในขณะที่ใน linux ถั่ว มี ปัญหาไดรเวอร์.

บริการระบบ

หลายคนไม่ชอบ systemd เมื่อมันเข้ามาในเคอร์เนล linux..สำหรับ Windows ฉันเริ่มคิดถึงมันแทน..Windows Services ไม่ใช่สิ่งที่มุ่งเน้นไปที่ผู้ใช้จริงๆ เช่น systemd Windows มี ตัวกำหนดเวลางาน เพื่อจัดการกับสิ่งที่ดูเหมือนone-shot ไฟล์หน่วย แต่อินเทอร์เฟซ (หรือขาดมัน) อีกครั้งนั้นน่ากลัว โชคดีที่เราไม่ต้องการ daemons แบบกำหนดเองใน windows เนื่องจากแอพพลิเคชั่น windows ส่วนใหญ่ที่เราใช้นั้นเป็นเพียง วิ่งเมื่อเริ่มต้น ตรรกะ และไม่ต้องการการกำหนดค่าขั้นสูงเพิ่มเติม อันที่จริงแยกขยะของเครื่องมือ ( ถอนหายใจ ) เพื่อให้ windows ทำงานเหมือนสภาพแวดล้อมสมัยใหม่ ดังนั้นเฉพาะแอปพลิเคชัน windows ดั้งเดิมอื่น ๆ ที่ใช้อยู่เท่านั้นคือเบราว์เซอร์ ( firefox ) เนื่องจากการเร่งความเร็ว GPU ของเบราว์เซอร์นั้นไม่ดีใน WSL (แม้แต่บน linux ดั้งเดิม...) และเครื่องเล่นวิดีโอ ( mpv )..และเกมส์แน่นอน...

แพ็คเกจ

ไม่ชอบ ช็อคโกแลต เนื่องจากต้องใช้สิทธิ์ของผู้ดูแลระบบ ฉันจึงมองหาแพ็คเกจ Scoop ก่อนเสมอ เนื่องจากมีการติดตั้งไว้ในโฟลเดอร์ผู้ใช้ ซึ่งสะดวกกว่าและสม่ำเสมอกว่า และทำให้การสำรองข้อมูลง่ายขึ้น

WSL/g

ฉันถูกกระตุ้นให้เปลี่ยนไปใช้ windows จากล่าสุด WSLg อัปเดต. ซึ่งต้องใช้ไดรเวอร์เมซ่าที่คอมไพล์ด้วยd3d12 การสนับสนุนแบ็กเอนด์สำหรับopengl . Windows มีเลเยอร์อูบุนตูแสดงตัวอย่างชุมชน แต่ฉันเลือกใช้ arch linuxเนื่องจากมีแพ็คเกจ AUR อยู่แล้วสำหรับ เมซ่ากับ d3d12 . เนื่องจากปัจจุบัน WSL ไม่รองรับ systemd ฉันจึงลงเอยด้วยการใช้ หัวหน้างาน เพื่อจัดการบริการบางอย่าง เพื่อให้แน่ใจว่าผู้บังคับบัญชาทำงานอยู่ มีการตรวจสอบในโปรไฟล์เชลล์สำหรับไฟล์ล็อคที่ควรสร้างขึ้นบนtmpfs ถ้าผู้บังคับบัญชาได้เริ่มดำเนินการมาก่อนแล้ว

WSL รองรับเท่านั้นext4 ระบบไฟล์ ในการใช้ระบบไฟล์อื่น คุณต้องเมาต์พาร์ติชัน (หรือดิสก์) ภายใน WSL VM โดยตรง อย่างไรก็ตามจะต้องติดตั้งด้วยตนเอง ในฐานะผู้ดูแลระบบ จากหน้าต่าง เพื่อทำให้สิ่งนี้เป็นอัตโนมัติเราสามารถใช้ตัวกำหนดเวลางานที่อนุญาตให้ข้าม UAC prompt โดยเรียกใช้งาน ด้วยเอกสิทธิ์สูงสุด . เราสามารถเรียกใช้ wsl--mount คำสั่งให้เมานต์ดิสก์ / พาร์ติชั่นที่เราต้องการแล้วเรียกใช้เมานต์ระบบไฟล์จากภายในลินุกซ์ เพราะของฉัน/home/ อาศัยอยู่ในภูเขาbtrfs ระบบไฟล์ฉันต้องเมานต์อัตโนมัติดังนั้นเราจึงใช้/etc/fstab เพื่อแมปพาร์ทิชันของเรา (byLABEL ) ถึง/home . เพราะเราใช้ nix กับnix-env เราจำเป็นต้องผูกเมานต์ร้านค้า nix ของเราเป็น/nix ในทุกการเริ่มต้น wsl และเนื่องจาก/tmp ไม่ได้เปิดtmpfs เราต้องสร้างการเมานต์โอเวอร์เลย์ซึ่งเมานต์tmpfs มากกว่าที่มีอยู่/tmp ไดเรกทอรี เพื่อรักษาไฟล์สำคัญ โดยเฉพาะไฟล์ X11 ที่จำเป็นในการพูดคุยกับเซิร์ฟเวอร์ X

การเมานต์เหล่านี้ดำเนินการด้วยสคริปต์ซึ่งควรรัน หลังจาก เราได้ติดตั้งดิสก์ภายในลินุกซ์แล้ว เราต้องการงานที่รันหลังจากงานเมานต์เสร็จสิ้น และเนื่องจาก WSL ใช้ VM ที่แตกต่างกันสำหรับผู้ใช้ windows ที่แตกต่างกัน สคริปต์จะต้องถูกเรียกใช้โดยไม่ใช้สิทธิ์สูงสุด (ไม่เช่นนั้นจะต่อเชื่อมบน VM ของผู้ดูแลระบบ)

เพื่อให้แน่ใจว่า WSLg ใช้งานได้ เราต้องแน่ใจว่า XDG รันไทม์DIR ถูกตั้งค่า เนื่องจากมันแตกต่างกันและตั้งอยู่ (โดยค่าเริ่มต้น) ที่/mnt/wlsg/runtime-dir . นี่คือลักษณะของหน้าต่างภายใต้ Wayland คุณจะเห็นได้ว่าเนื่องจากผู้แต่งหรือแอปพลิเคชันสามารถวาดการตกแต่งหน้าต่างเวย์แลนด์ได้ พวกเขาจึงคงการกำหนดค่าของชุดรูปแบบ GTK/QT ของคุณ ดังนั้นคุณจึงจบลงด้วยหน้าต่างค้นหาธีม linux ดั้งเดิม ภายในเปลือกของ Windows...ซึ่งในตอนแรกจะสับสนเล็กน้อย

เทอร์มินัลอีมูเลเตอร์

บน linux ฉันใช้ คิตตี้ เนื่องจากเป็นหนึ่งในเทอร์มินัลอีมูเลเตอร์รุ่นใหม่ที่มีการเร่งความเร็ว GPU[2] และโหมด daemon ที่ค่อนข้างเสถียรซึ่งอนุญาตให้ใช้หลายหน้าต่างที่มีอินสแตนซ์เดียวกัน อย่างไรก็ตามใน windows ฉันขลุกอยู่กับเทอร์มินัลที่ใช้ windows บางตัวเช่น Windows Terminal, เทอร์มินัลคล่องแคล่ว, wsltty, conemu แต่ในที่สุดฉันก็เปลี่ยนไปใช้ wezterm . แม้จะเป็นเทอร์มินัลที่ใหม่กว่า แต่ก็มีคุณสมบัติทั้งหมดที่คุณต้องการจากเทอร์มินัลอีมูเลเตอร์:

  • แท็บและบานหน้าต่าง โดยที่คุณไม่ต้องการ tmux

  • มัลติเพล็กซ์ (โหมดหัวขาดระยะไกล) โดยที่คุณไม่ต้องการ tmux

  • โหมดคัดลอกข้อความพร้อมการโยงปุ่ม vi คุณไม่จำเป็นต้องมี tmux

  • เลือกด่วน โหมดคำใบ้เพื่อเลือกวัตถุข้อความอย่างรวดเร็วจากการเลื่อนย้อนกลับ

  • คำสั่งเริ่มต้นสำหรับแท็บใหม่

  • การปรับแต่งคีย์แมปได้ดีมาก นอกเหนือจากข้อเท็จจริงที่ว่าคุณไม่สามารถใช้ Shift/Ctrl/Alt ซ้าย/ขวา/Ctrl/Alt เป็นตัวปรับแต่งต่างๆ ได้

ด้วย Windows ฉันยังปรับเปลี่ยนเวิร์กโฟลว์ของฉันเล็กน้อย โดยที่ฉันไม่ต้องการเทอร์มินัลที่มีโหมด daemon ที่รองรับหน้าต่างหลายบานแยกจากกัน โดยใช้เทอร์มินัลดรอปดาวน์ เพื่อให้ wezterm หล่นลง หลังจากลองใช้สคริปต์ AHK แบบเลื่อนลงมาหลายตัว ในที่สุดก็พบว่า หนึ่ง ที่ได้ผลค่อนข้างดี ทางเลือกที่ดีกว่าจะเป็นwindows-terminal-quake แต่ฉันมีปัญหากับมันทำงานผิดปกติกับเดสก์ท็อปเสมือนที่จัดการโดย dexpot ในขณะที่ mitty-quake-console ดูเหมือนว่าจะเข้ากันได้กับเดสก์ท็อปเสมือน dexpot เป็นไปได้มากว่าเมื่อประสบการณ์เดสก์ท็อปเสมือนของ Windows ในตัวดีขึ้น ฉันจะเปลี่ยนไปใช้ windows-terminal-quake WezTerm เป็นแพลตฟอร์มข้ามแพลตฟอร์มด้วย ซึ่งหมายความว่าฉันสามารถใช้งานต่อกับการติดตั้งลินุกซ์แบบเนทีฟได้ เป็นการค้นพบที่โชคดีมากสำหรับสวิตช์ Windows + WSL นี้ เนื่องจาก Kitty ยังไม่รองรับ Windows และตัวเลือกเทอร์มินัลอื่น ๆ มีความรำคาญค่อนข้างมาก ฉันทำใจไม่ได้กับ.
