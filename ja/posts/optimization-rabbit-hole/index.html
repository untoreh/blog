Pythonの最適化ラビットホール

freqtrade 暗号取引ボットのバックテストを高速化します。

フリートレード Python の暗号取引ボットです。私はしばらくフォークで遊んで、いくつかのバグや気に入らない動作を修正していました。ここでバグを列挙するとつまらないので、代わりに私が面白いと思うことについて話します。

freqtrade (FQT) の仕組みの概要

FQT はイベント ベースではなく、ループ ベースです。これは何を意味するのでしょうか？エグゼキューター (ライブ オペレーション) またはバックテスターに​​ついて話している可能性があります。この場合、それらは両方ともループベースです。

ここでちょっと寄り道。私は何が好きですか？ループまたはイベント?

  • ループは一般に、イベントに比べて単純で推論が容易です。イベント OTOH は、現実世界をより適切にモデル化できます。また、ループはイベントに比べて並列化が容易であり、一般的に高速に実行されます。

  • ループ ベースのバックテストは、ライブ エグゼキューターと並行して実行できます。これは、後方評価がトレーディング戦略の一部になることを意味します。イベント ベースのバックテスト システムの実行 住む 私には混乱のようです。イベントには時間の概念が必要ですが、ライブ取引中は現在しかありません。では、何をどのようにシミュレートしますか?

...FQT に戻ります。ライブ モードでは、ループで注文を処理します。 X 秒ごとに更新の交換をクエリします。以下を追跡します。

  • すべての資産のすべての最終価格を保持するペアリスト (それをサポートする取引所用)

  • 新鮮な OHLCV データ

  • オープンオーダーのステータス

ルロー

ループベースのライブ実行には明らかな問題があります。同期です。多くのペアを処理すると、実行の応答性が低下し、反復回数が減り、価格アクションの展開に遅れが生じます。そのため、バックテストにはループを好みますが、ライブ実行にはイベントを好みます。

欠点は？一方にイベントを使用し、もう一方にループを使用すると、戦略のモデルまたはパラメーターが分岐しやすくなり、現実と一致しなくなります。

これに対する私の反論は、自己評価 (バックテスト) とライブ実行を常に実行することで、この問題が回避されるということです。 最終的 整列します。

FQT には多くの構成パラメーターがあり、多くの場合、ボットによる注文の実行方法を戦略が制御できないため、ボットと戦略の区別が灰色に変わります。ボットは、テイク プロフィットの価格制限のグリッドである「roi」と「トレーリング ストップロス」を同じように処理しますが、マイナス面があります。戦略はパラメーターを提供するだけで、残りはボットが行います。私はこれのファンではありませんでした。

売買シグナルのコールバック。取引は、データフレームでシグナルを計算するコールバックによって返されたシグナルに基づいて行われます。これに関する問題は、信号が静的であることです。 1行のデータフレームを動的に計算し、それをボットにフィードすることはできますが、バックテストはできません.バックテストをそれほど重視しない場合は、これで問題ありません。

の探求 ベクトル化された バックテスト

高速化するために、多くのバージョンのバックテストを試しました。それだけの価値はありましたが、まったくありませんでした。 numpy 配列を使用した売買操作の間のロジックの計算。脳でテトリスをするようなものです。ブロックがあなたの灰白質で作られている場所を意味し、あなたはそれらをまとめようとします.ここにそれらをリストします

  • チャンクバックテスト .この例では、numpy 配列のみを使用して取引を折りたたもうとします。売買、シグナル、ストップロス、トレイリング ストップロス、ROI グリッド計算、時間加重 ROI 計算の実行の正しいフローの間でジャグリングします。すべてでこぼこした配列で、かなり混乱していました。

  • VBT ベース このバージョンは活用しようとしました ベクトルbtバックテストを実行する numba ベースの python ライブラリ。驚いたことに、numpy バ​​ージョンよりもそれほど高速ではありませんでした。これは、VBT 互換の型への多くの型変換が必要だったため、実行によって得られた速度がそこで失われたためです。

  • 範囲のループ numbaを自分で使うのはこれが初めての試みでした。 numba は numpy 関数をネイティブに呼び出すことができるため、numpy ベースのバックテスト フレームワークを numba で動作するように調整しました。ゲインは非常に小さかった。すべてを numpy 配列に入れるコストがボトルネックでした。 numpy 変換後、numpy の実行は既に高速であるため、その後に得られる利益は最小限でした。

  • ろうそくのループ これは当然のことながら最後のものでした。最高のパフォーマンスを発揮したのは実装でした。これは、ろうそくに対する「単純な」反復であり、すべて numba jitted 関数内で実行されたためです。コンセプトは単純でしたが、numba バグを処理しながらすべての ROI とストップロス ロジックを実装するのは良くありませんでした。最終的には非常にうまく機能しましたが、私が登らなければならなかったのは、numba の非互換性の山の後でした。

また、バックテストにいくつかの追加機能を追加しました。 スプレッドと流動性 計算、および前述の時間加重 ROI。

の探求 平行 最適化

高速バックテスターが必要な理由私はそれらの多くを実行して、最高のパラメーター構成を見つけることができるようにしたいので...もちろん、過剰適合、過剰パラメーター化、焦点の欠如などの概念を無視して...リストしましょう私が取り組んだこと：

    ベイジアン最適化に基づかない別のオプティマイザを使用できます。 これです .または、N 個の独立したオプティマイザーを実行して、各プロセス/オプティマイザーが独自の履歴を使用して新しいジョブをクエリするか、オプションで他のすべてのオプティマイザーと共有されている履歴を使用します (ただし、これにより、しばらくするとすべてのオプティマイザーが収束します。プロパティの場合とそうでない場合があります)。

    すべてのプロセス間で観察結果を共有することは、完全な災害でした。 float の大きなリストの pickle 化と unpikling は非常に遅くなります。検索が進行するにつれて、観測の数が増え、プロセス全体がますます遅くなり、観測の評価よりもシリアル化に時間がかかるようになります。マネージャー プロセス、プレーン ファイル、またはメモリ マップされたファイルのいずれを使用しても遅かった。最高の お金のために強打 したがって、アプローチは、個別のオプティマイザを実行し、それらの間で観測を共有せず、すべてをメモリに保持することであることが判明しました。

  • 実行された最適化の視覚化。 FQT には、最適化を実行するための優れた視覚化が既にありました。私はそれを改善しました。作成する必要がありました 物体 すべての観測を表す。次に、次のような追加の後処理を実行できます 正規化 と フィルタリング

  • にとって 交差検証 データを分割し、さまざまな範囲でバックテストを適用する必要があります。最適化状態の追跡における私たちの努力のおかげで、交差検証からの出力を別のより洗練された最適化プロセスのシードとして使用することができました...もちろんオーバーフィットするために.
