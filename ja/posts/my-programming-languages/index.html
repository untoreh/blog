私のプログラミング言語

私が選んだプログラミング言語

リストを短くするために私が心がけているのは、 エコシステムごとに 1 つの言語

ジュリア

これはエコシステムではありませんが、私が最も選択する言語であるため、最初に選択する必要があります。これが私のお気に入りの理由は、それが唯一の動的言語であるためです。 本当に人間工学に基づいた 、ベンチマークが簡単にできる点、Makie、型の構成方法、ディスパッチ システムがパラメトリック型とシームレスに統合される方法が気に入っています。

ジュリア 速度と使いやすさのためにメモリとコンパイル時間の一部を犠牲にしており、最大 500M の RAM より小さいプロセスを見ることはほとんどありません。したがって、Electron や Java が嫌いな場合は、julia でも問題が発生する可能性があります。小さな julia バイナリを提供するソリューションはありますが、それらはすべて実験的なものであり、とにかく julia は動的な第一言語であるため、主なターゲットのユースケースは次のとおりであると思います。常に repl から実行する必要があり、コンパイルされたバイナリは常に第 2 市民になりますが、時間が経てばわかります。

最近、mojo と呼ばれる言語が登場し、これが julia に対する「脅威」ではないかと疑問を抱く人も出てきました。かなり長い間 julia を使用していて、(私と同じように) mojo のドキュメントをざっと読んだことがある人なら、mojo が静的言語であることに気づいたかもしれません。 ボルト締めされた 動的な Python の上にあるため、Python と「統合された」言語とは言いがたいため、リンゴとオレンジを比較していることになります...

julia を他のものと比較したい場合は、他の「JAOT」ベースの言語がいくつかあります。

  • numba: julia llvm jit と同じ、Python、numpy、numba 固有のイディオムのミックスマッシュに基づいています。これに基づいて大規模なプロジェクトを実際に構築することはできません。小さな関数の高速化にのみ役立ちます。

  • gccemacs: ネイティブ コンパイル モードの emacs は julia と同じことを行いますが、はるかに弱い形式です。lisp から非常に具体的な型を実際に推論できないため、間違いなくそうではありません。 emacs リスプ 、gcc jit に基づいています。

ニム

これを「C」エコシステムと呼ぶことができます。私の 2 番目に好きな言語は ニム 、しかしツールがクソなのでめったに使いません。ですから、誰か nim ツール開発に数百万ドルを投じてください、ありがとう!これが私のお気に入りです。なぜなら、前に述べたように、julia は出荷するのが非常に難しく、ギガバイト サイズのバイナリを出荷したくないからです。代わりに、Nim は非常に小さなバイナリをコンパイルし、それを静的にすることもできます。

(言語の観点から見た) 唯一の問題点は、スレッドと非同期の間の相互作用です。非同期クーリンとスレッドの間での受け渡しには落とし穴がたくさんあるため、スレッドと非同期ランタイムの間でのやりくりを容易にする何らかのシステムが欠けています。また、非同期ランタイムを使用すると、非同期クロージャの書き換えなどによりバイナリが非常に大きくなる可能性があります。要するに、バイナリにはコード生成された関数が大量に含まれており、関数の命名スキームのせいでデバッグも困難になります。

全体的に出荷が簡単であることに加えて、優れた ffi サポート、ドット呼び出し構文、Lisp よりも優れたマクロ/テンプレート システム (sexprs の不足を無視する場合)、julia のマクロは問題ありませんが、nim はマクロを「ディスパッチ」できます型よりもテンプレートを使用することで、「汚れと衛生」の間で選択できるようになり、言語サーバーは、関数、マクロ、テンプレートの間で、Rust ですら達成できなかった多くのインテリジェンスを提供できました (2 秒ごとにクラッシュしなければの話ですが)。そして死ぬほどゆっくりしてください…ため息）。

Nim には nimscript もあります。私の願いは、ホット コードのリロードが再び取り上げられるようになったら、それを中心に本格的なインタラクティブ repl を構築できることです。

さび

nim ツールがクソなので私が使っているのは Rust ですが、nim ツールがクソじゃなかったら nim を使うでしょう。私は個人的には、Rust メモリの安全性が nim よりもそれほど優れているとは思いません。 Nim には「大胆な同時実行性」がありません。つまり、Rust と比較すると、スレッドを処理するときに Nim でセグメンテーション違反が発生しやすいのです。ただし、Rust はスレッドの安全性を達成するためにライフタイムを使用しますが、nim には言語レベルでそのようなものがありません。ライフタイムはおそらく Rust 言語の機能の中で最もわかりにくく、読みにくいため、そこに理由があるかもしれません...

一般に、ある時点以降はメモリのバグよりもビジネスのバグの方が一般的になるため、言語自体の可読性を損ない、複雑さを増大させながら、メモリの安全性の利益を逓減させるために言語を過度に複雑にすることは望ましくないと言えます。言語によってビジネスのバグがどの程度発生しやすくなるかを定量化することはほぼ不可能であるため、この点は常に議論の余地があります。

残り

これらは、私が考えている「代替」言語のリストの上位にある言語にすぎません。 そうかもしれない エコシステムごとに 1 回選択します

  • Beam: gleam と elixir のどちらを選択するかは難しいです。最終的に gleam を選択するか、両方を試してどちらかを選択することになるかもしれません。もちろん、これらは根本的に異なる言語ですが、それがエコシステムごとに 1 つだけを選択することになると述べました。

以下は、これらのエコシステムを私が使用するものですが、実際には試していません。

  • JVM/Graal: Clojure。過去には僅差で 2 番目に scala があったかもしれませんが、事実上「工業用」Lisp は Clojure だけなので、これを選ぶ必要があります。

  • Node/Deno/Bun: typescript を選択するのは、それが最も一般的なコンパイルから js 言語であるという理由だけです。将来、ReScript のエコシステムが十分に大きくなったら、ReScript を検討するかもしれません。

  • CLR: 何もない?私は F# と言いますが、CLR はないと言いたいです。 clr には clojure と swift の実装もあるようなので、それはありますが、私の印象では、これらの「ポート」のいずれかを使用すると、多くの非互換性に遭遇するだろうということが常にありました。とにかく、自分が clr に近づくことはないと思います。個人的な選択。

  • Lisp: Lisp はエコシステムですか?ええと、そうではありません。ただし、これはすべて sexpr であり、あるスキームから別のスキームに移植することは、検索と置換のクエリに近いと言う人もいるかもしれません。 emacs lisp に gccemacs を活用して高速コードを吐き出す高速な方言があれば、私は common lisp よりもそれを強く検討するでしょう。代わりにスキームを選択しなければならないとしたら、それは問答無用で迷惑でしょう。実際のソフトウェアを Commonlisp または Racket で作成していると思いますか?可能性は低いです。もし Lisp が必要なら、clojure を選ぶでしょう。

これらの「エコシステム」言語の選択とは別に、名誉ある言及に値する言語が他にもいくつかあります。

  • raku: 私は過去にある程度の PHPing をやったことがあり、perl はまったくありませんでしたが、言語の比類のないコードゴルフ容易性がなければ、raku は本当に興味深いと思います。

  • 赤: フルスタックの約束は魅力的であり、Rebol の伝統は十分にユニークなので、適切なサイズのアプリを書いて試してみる価値があります。

  • Dart: UI 開発につまずいたら、それを試すかもしれませんが、たとえ UI 開発であっても、自分のトップ言語の 1 つを使い続けたいと考えています。すると、次のような質問になります。 どのくらいで dart (または haxe!) はマルチプラットフォームサポートのおかげで生産性を向上させますか?

  • Verse: 壮大なゲームの言語。プロローグ風の言語に対する事実上唯一の現代的なアプローチであるため、ユニークです。誰もが UE ブループリントを嫌っているため、この叙事詩がこれを構築したのは明らかだと思われます。しかし、言語が UE と密接に結びついたままであれば、私はそれを試すことはないと思います。

ocaml や haskell などのより機能的な言語や、zig や vlang などのより単純な言語など、他にも人気のある言語はたくさんありますが、これらはあると思うので言及しませんでした。 デッドセットすぎる パラダイム (不変性/単純さ) よりも、一般的に、私はより「丸い」言語を好むと思います。

このリストはそれほど短くはありませんでしたが、申し訳ありません。
