Pikiran babagan skrip cangkang

Bash/ zshskrip cangkang rampung tugas. Iki uga elek banget. Mungkin iku dimaksudake. Kurang sampeyan kepengin ndeleng skrip kasebut bakal luwih cekak. Aku dudu penggemar nggunakake bash / zsh minangka cangkang interaktif, jumlah plugin sing dibutuhake kanggo nggawe pengalaman sing apik banget, lan saben plugin tambahan minangka jebule kemungkinan mudhun alon-alon cangkang. Aja nyuda jumlah gaweyan sing ditindakake dening psprint karo zinit & kanca nyengsemake banget (lan uga minangka siji-sijine pilihan sing bisa dingerteni kanggo manajer plugin yen sampeyan pengin nggunakake zsh minangka cangkang interaktif). Nanging ora bisa ngatasi kasunyatan manawa kabeh plugin sing dimuat kasebut ora dituduhake memori ing antarane (mungkin) akeh proses cangkang.

Cangkang iwakminangka cangkang sing paling disenengi nalika nulis. Jumlah plugin sing diinstal minimal banget, amarga umume alat sing sampeyan pengin saka cangkang metu saka kothak . Skrip luwih alon tinimbang zsh nanging ora ana sing peduli karo kacepetan skrip dadi ora apa-apa. Sintaksis kasebut minangka warisan ML (kaya Julia) kanthi pratelan pungkasan, lan dudu kurung kriting. Nulis skrip luwih akeh verosa tinimbang bash / zsh, nanging uga luwih bisa diwaca. Yaiku kayata substitusi proses, interpolasi strings lan piping pungkasane dadi verosa. Wis global variabel, dituduhake ing kabeh proses sing mlaku, lan universal variabel, dituduhake ing sesi, sing nggawe perkara luwih gampang tanpa kudu nulis kabeh lan nggawe skrip sing lali.

Elvish yaiku cangkang kanthi dhukungan kanggo akeh perkara, kayata mode daemon utawa web sing ngidini ngirim perintah saka jarak adoh ...? Aku ngira akeh banget lan aku mula ngarani kembung. IMHO cangkang kudu ngupayakake kesederhanaan, lan bisa mandhiri. Aku ngira cangkang kudu dirancang kanggo ngatasi skenario nalika pangolahan ngasilake kaping pirang-pirang, tumindak kanthi konsisten wiwit wiwitan nganti pungkasan lan bisa mandhiri supaya bisa ngatasi kacilakan kanthi anggun. Dadi, sanajan kabeh fitur liyane (kaya data sing wis kabentuk) ... daemon ora bisa ditrapake kanggo aku.

Cangkang Ion yaiku meh posix cangkang saka redox, sing uga digunakake ing unix, kanthi tujuan ugadigunakake ing windows, sanajan ora bisa ATTOW. Tujuane yaiku supaya dadi cangkang sing luwih apik, mula entuk sing apik saka posix lan ditambahake. Kayane fokus kanggo nyedhiyakake basa skrip sing apik, nanging dudu pengalaman interaktif sing apik banget, cangkang iwak wis ngrusak aku, lan aku ora bakal bali menyang cangkang tanpa kelas siji interaktivitas

Nihan minyak nduweni tujuan kanggo ngganti bash, sing bisa dipuji, nanging target mung sistem unix sing ora ana sing katon ing Windows. Wis fokus kanggo mbenakake lapisan kompatibilitas bash nganti saiki, kanthi rencana mbesuk kanggo fokus ing bagean interaktivitas. Ngelingi kasunyatan manawa bagean interaktif cangkang durung ditulis, nggawe aku ora mangu-mangu babagan mbesuk lan sanajan dakkarepake ora, aku ora bisa ngeculake sadurunge entuk target sing dituju. Uga kekurangan dhukungan windows ora cocog.

Rash cangkang kanthi dhukungan kanggo data terstruktur sing bisa digunakake Raket obyek, nalika aku nggoleki raket, aku bakal nyoba, sanajan pengalaman sadurunge karo [ emacs Nihan ] pancen ora apik, wiwit eshell mung bisa digunakake ing emacs, lan ora bisa digunakake kanthi apik cangkang sistem ... kajaba sistem sampeyan minangka emacs (sanajan emacs minangka editor utama lan aku nulis sawetara lisp, dudu sistem operasi saya). (Ing kene aku uga kudu nyebutake Julia REPL, replika sing paling apik kanggo basa pamrograman).

Kekuwatan Powershell, cangkang Windows utama uga dianggo ing linux lan macs, amarga platform silang NET . Nanging tembung kasebut lambat, alon, lan bisa uga (ora diverifikasi) diisi windowsisme ; Iki ora gedhe kanggo aku.
