script de shell

Reflexões sobre script de shell

Bash/ zsho shell scripting dá conta do recado. Também é muito feio. Talvez seja intencional. Quanto menos você quiser examiná-lo, mais curtos serão os scripts. Não sou fã de usar bash / zsh como um shell interativo, a quantidade de plug-ins necessários para compensar uma experiência agradável é assustadora e cada plug-in adicional é uma armadilha para possíveis lentidões do shell. Não obstante, a quantidade de trabalho realizado pelo psprint com zinit e amigos é muito impressionante (e também meio que a única escolha sensata para um gerenciador de plugins se você quiser usar zsh como um shell interativo). No entanto, isso não resolve o fato de que todos os plug-ins carregados são não compartilhado memória entre (provavelmente) muitos processos do shell.

Concha de peixeé meu shell preferido no momento em que escrevo. A quantidade de plug-ins instalados é mínima, porque ele faz a maioria das coisas que você gostaria de um shell sai da caixa . O script é um pouco mais lento do que zsh, mas ninguém se preocupa com a velocidade do script, por isso está bem. A sintaxe é boa para a herança de ML (como Julia) com instruções finais, em vez de chaves. Escrever scripts é mais detalhado do que bash / zsh, mas também é muito mais legível. Coisas como substituição de processos, interpolação de strings e tubulação acabam um pouco mais prolixas. Tem global variáveis, compartilhadas por todos os processos em execução, e universal variáveis, compartilhadas entre sessões, o que torna as coisas mais fáceis sem ter que escrever tudo e criar scripts esquecidos.

Élfico é um shell com suporte para várias coisas, como um daemon ou modo web que permite enviar comandos remotamente ...? Acho que é demais e começo a chamar de inchado. IMHO um shell deve se esforçar pela simplicidade e ser independente. Acho que os shells devem ser projetados para lidar com cenários em que são processos gerados uma quantidade notável de vezes, que se comportam de forma consistente do início ao fim e que são autocontidos de forma que possam lidar com travamentos de maneira adequada. Portanto, apesar de todos os outros recursos (como dados estruturados) ... daemons não servem para mim.

Ion Shell é quase posix shell do redox, que também funciona em unix, com o objetivo de tbtrabalhando no Windows, embora não funcione ATTOW. Seu objetivo é simplesmente ser uma casca melhor, por isso pega o bem do posix e adiciona a ele. Parece estar focado em fornecer uma boa linguagem de script, mas não uma experiência interativa muito boa, a concha de peixe me estragou e não vou voltar a uma concha sem primeira classe interatividade.

Reservatório de óleo tem o objetivo de substituir o bash, o que é louvável, no entanto, ele visa apenas sistemas unix sem Windows à vista. Ele se concentrou em consertar a camada de compatibilidade do bash até agora, com planos futuros para se concentrar na parte da interatividade. Considerando o fato de que a parte interativa do shell ainda não foi escrita, me deixa um pouco em dúvida sobre seu futuro e, embora eu espere que não, eu acho que pode perder força antes de atingir seus objetivos pretendidos. Além disso, a falta de suporte para janelas não é o ideal.

Irritação na pele um shell com suporte para dados estruturados que funciona com Raquete objetos, quando eu olhar para a raquete mais eu vou tentar, embora minha experiência anterior com o [ shell emacs ] não é muito bom, já que o eshell só funciona dentro do emacs, e não funciona bem como um shell do sistema ... a menos que seu sistema seja emacs (apesar de emacs ser meu editor principal e eu escrevo algum lisp, não é o meu sistema operativo). (Aqui devo também mencionar o Julia REPL, o melhor repl para qualquer linguagem de programação).

Powershell Powershell, o shell principal do Windows também funciona em Linux e Macs, graças à plataforma cruzada INTERNET . Mas é muito prolixo, lento e provavelmente (não verificado) cheio de windowsism ; É um grande não para mim.

Nu shell... é claro que deixamos o melhor para o final. Esta é provavelmente a minha próxima tentativa de shell; atenção de primeira classe para interatividade, consideração cuidadosa de cada shell integrado e de plataforma cruzada e texto estruturado.

Shells que não estão nesta lista ATTOW provavelmente não foram considerados interessantes o suficiente.
