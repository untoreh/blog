Construindo um agregador de conteúdo para diversão e lucros?

Um aplicativo completo que extrai, processa e apresenta conteúdo da web... na web.

Por quê?

Sobrecarga de informação ? É meio ruim hoje em dia, muitas fontes de informação de baixo sinal para ruído, restringir seus "feeds" de forma que você não fique sobrecarregado é meio difícil. Uma ferramenta que filtrasse as informações e as apresentasse em um formato fácil e rápido de digerir seria muito útil. É por isso que considero agregação de conteúdo um campo perene para perturbações. É e será sempre (enquanto a internet for gratuita e houver liberdade de expressão) uma boa oportunidade de negócio. É uma daquelas instâncias em que tudo gira em torno da execução (e nada da ideia).

Gerenciando expectativas

Dito isso, meu aplicativo no final não faz nenhuma filtragem real. Na verdade, simplesmente agregados conteúdo da web. Isso ocorre porque não criei usuários nele e há pouco incentivo para filtrar se não puder ser personalizado por usuário.

O Raspador

A raspagem é feita em... você adivinhou, python. No entanto, nenhum módulo de "raspagem" ad hoc é usado.

O que você raspa?

Decidir o que raspar depende da categoria do conteúdo. Chamamos as categorias de "tópicos".

  • Cada tópico tem uma lista de palavras-chave.

  • A lista de palavras-chave, se extraída do google adwords usando seus api python

  • As palavras-chave são consultadas em vários mecanismos de pesquisa, em uma ordem round robin periodicamente. Se a instância tiver vários tópicos, os tópicos com menos conteúdo disponível serão pesquisados ​​primeiro. Para realizar as pesquisas em que contamos procurar com procurações. O Searx não é exatamente compatível com bibliotecas, já que seu uso principal é para o front-end, portanto, foi necessário pesquisar o processo correto para inicializar o módulo para realizar consultas. Para acelerar as coisas, use o threadpool para executar várias consultas simultaneamente, usaremos o threadpool com frequência ao longo do projeto.

  • Cada pesquisa por palavra-chave gera uma lista de fontes de conteúdo em potencial (os resultados dos mecanismos de pesquisa) que são salvas no armazenamento para processamento posterior.

  • Quando queremos encontrar um novo conteúdo para um tópico específico, primeiro verificamos se existem fontes disponíveis, caso contrário, geramos novas fontes a partir da lista de palavras-chave.

  • As fontes são processadas por meio de duas bibliotecas trafilatura é o principal, se falhar, fazemos backup para ganso . Também tentamos encontrar feeds para links adicionais (que seriam considerados novas fontes). Para feeds, usamos localizador de feeds mas uma simples análise do html para rsslink tags também seriam suficientes.

  • Nosso principal tipo de conteúdo é umArticle , que do lado do python é apenas um dict, do lado do nim com análise como um objeto. Chaves:

    • title : o cabeçalho do artigo

    • content : o próprio artigo. Para determinar o que é um bom artigo, passamos por diferentes etapas de filtragem:

      • Primeiro, verificamos se a trafilatura ou o ganso têm texto e se é longo o suficiente. Nosso tamanho mínimo é de 300 palavras. Se o tamanho não corresponder, descartamos a fonte (não retornamos nada).

      • Em seguida, buscamos o título e o limpamos removendo urls e espaços em branco

      • Se o idioma for estrangeiro, traduzimos de volta para o inglês (normalizamos para o inglês) tanto o conteúdo quanto o título.

      • Neste ponto, verificamos palavrões usando profanity_check . Não que a verificação de palavrões seja baseada em inglês para a tradução anterior é necessária. Caso contrário, precisaríamos de um modelo de palavrões para todos os idiomas.

      • Depois de substituir palavrões usando filtro de palavrões, continuamos limpando o conteúdo. Verificamos se o artigo é relevante. As regras que usamos são:

        • O conteúdo deve começar com caracteres alfanuméricos, caso contrário há uma alta variação de que é lixo.

        • Tanto o título quanto o conteúdo não podem ser "ruído". O ruído é definido por um regex que captura palavras-chave como "login", "inscrição", "acesso negado"... etc.

        • Pelo menos uma palavra no título deve estar presente no corpo. Caso contrário, é possível que a análise tenha escolhido as partes erradas da página de origem para o conteúdo.

      • Se os testes de relevância forem aprovados, como etapa final, limpamos o conteúdo contra ocorrências de muitos colchetes, espaços em branco, caracteres repetidos e caracteres especiais.

      • Se a limpeza não excluiu tudo, continuamos processando o artigo.

    • source : o link apontando para a fonte original que analisamos

    • lang : a linguagem do artigo, usamos língua para detectar o idioma

    • desc : o resumo, caso contrário, um trecho do conteúdo

    • author : o autor, caso contrário, o título da página inicial do sourcelink

    • pubDate : a data de publicação do artigo, ou agora

    • topic : o tópico ao qual este artigo pertence

    • tags : palavras-chave relevantes para um artigo, usamos a biblioteca de extração kw mais rápida, que é ancinho , as alternativas consideradas são pyate (combo básico), classificação de texto e máquina de frase

    • imageTitle : o texto alternativo para a imagem

    • imageOrigin : se a análise da fonte (para imagens usamos moça ).

    • imageUrl : o link real para a imagem. Usamos uma verificação de filtro bloom para imagens duplicadas, porque não gostamos de duplicatas.

    • icon : o favicon do link de origem

  • Depois de processarmos uma palavra-chave, salvamos seus artigos e feeds encontrados no armazenamento. Eles serão usados ​​pelo editor.

A raspagem acontece continuamente, é um demônio. O pseudocódigo do loop principal se parece com isso e está configurado por local:

  • sincronizar proxies para sempre

  • para cada tópico classificado por contagem de (artigos) não publicados (de baixo para cima), faça o seguinte

    • se o intervalo mínimo do último trabalho for passado, execute um trabalho de análise para o tópico. O intervalo aumenta quanto mais artigos não publicados tivermos para um tópico e será sempre 0 se não tivermos artigos não publicados.

    • Faça o mesmo, mas para feeds (que coletamos de fontes, se houver)

    • Se o site for criado para criar novos tópicos, crie um. (Isso só faz sentido se você não decidir a lista de tópicos na criação do site.)

    • Escolha um artigo entre os publicados e envie um tweet (enviamos 3 tweets por dia, usando python-twitter)

    • Escolha um artigo entre os publicados e atualize uma página do facebook (fazemos 1 atualização por dia, usando facepy)

    (Também conectamos o reddit, mas o reddit não permite cross-posting, então foi um esforço desperdiçado.)

Publicação

Assim que tivermos conteúdo para publicar, temos que decidir o que publicar e com que frequência. Não inventei nenhum truque aqui, porque, como mencionei anteriormente, a escolha do que exibir depende do usuário. Então, nós apenas publicamos de mais recente para o mais antigo , com o raciocínio de que algo que raspamos mais recentemente é mais relevante, é um LIFO fila. A publicação, embora bastante dependente de ferramentas python, ocorre em nim, porque é executada de forma adjacente ao servidor, que também está em nim.

A lógica de publicação

A publicação acontece continuamente, e assim como a raspagem tem intervalos ociosos, a publicação também, de forma inversa à raspagem. Quando raspamos, desaceleramos quando temos um longo O suficiente cache de artigos inéditos, com a publicação desaceleramos quando nosso cache começa a diminuir muito. Desta forma, juntamente com a raspagem, deve haver sempre algum conteúdo a ser publicado às vezes no futuro. A lógica de publicação real:

  • Busque um lote de artigos não publicados do cache. (Optamos por publicar 3 novos artigos por execução.)

  • Verifique se são duplicados. A verificação de dup é feita por meio de hash sensível à localidade, aproveitando um nim lib, minhash . O LSH é bastante intensivo em CPU (você sabe ... hashing) e requer seu próprio thread (há algumas outras tarefas com as quais lidamos que requerem seus próprios threads).

  • Renderização da página: isso não é necessário, pois o servidor lida com as consultas em tempo real, mas a renderização aqui é uma forma de pré-cache.

  • Manipulação de página. Como estamos lidando com um site, temos que escolher quantos artigos exibir por página e incrementar as páginas à medida que publicamos mais artigos. Optamos por agrupar artigos em páginas de ~10. A página mais recente tem sempre menos de 10 artigos.

  • Salve o estado dos artigos publicados, isso significa mover os artigos do status "não publicado" para "publicado" e o banco de dados LSH.

  • Depois de publicar novos artigos, temos que limpar os caches obsoletos. Temos que limpar a página inicial, a página do tópico, o mapa do site e o feed rss.

Servimos 11 tipos diferentes de URLs:

  • página inicial: extrai artigos dos tópicos mais recentes, pseudo aleatoriamente, não fazemos nenhuma classificação com base na popularidade.

  • ativos genéricos (sob/assets/ path): são mapeados diretamente para um diretório dedicado

  • imagens genéricas (em/i/ pah): procuramos imagens externas para gerar tamanhos que se ajustam ao nosso site responsivo, quando as imagens não estão disponíveis, um pixel transparente ou um ícone de imagem é exibido como padrão.

  • arquivo robots.txt

  • sitemaps (para homepage e tópicos e páginas): A homepage hospeda o sitemapindex apontando para todos os tópicos sitemaps, os tópicos sitemaps apontam para todas as páginas do tópico, as páginas sitemap apontam para todos os artigos da página.

  • pwa manifest: o pwa manifest deve permitir que o site seja instalado como pwa (mas francamente não testei isso)

  • pesquisas: A pesquisa aproveita sônica com o psônico ligações.

  • sugestões: As sugestões também são tratadas por meio de bibliotecas sônicas. Mas eles exigem

  • feeds: Como os mapas do site, temos feeds diferentes para a página inicial e para os diferentes tópicos, embora não haja feeds para páginas individuais por razões óbvias.

  • páginas de tópico: A página dedicada a cada tópico (por exemplo, com caminhodomain.com/my-topic/ ) Extrai os últimos artigos publicados para o tópico, que pertencem a um inacabado página.

  • páginas de artigos: A página do artigo mostra o título do artigo, descrição, link da fonte, tags, horário de publicação (no rodapé) e na parte inferior puxamos 3 artigos relacionados. Os artigos relacionados são buscados usando uma consulta de pesquisa no título ou tags do artigo.

Ligações do Python para nim tem que descartar objetos python. O problema é que temos que controlar quando o nim faz o GC. A biblioteca nimpy assume que o GIL está sempre bloqueado (ele o bloqueia no início), para que seja livre para chamar o python a qualquer momento. Mas nós desbloqueamos o gil para permitir que um threadpool do python execute o código enquanto o nim executa outras coisas. Se o python GIL estivesse sempre bloqueado por nim, o threadpool ficaria ocioso na maior parte do tempo.

proc pop*[K, V](t: AsyncTable[K, V], k: K): Future[V] {.async.} =
  var popped = false
  withLock(t.lock):
    if k in t.table:
      popped = t.table.pop(k, result)
  if not popped:
    if k notin t.waiters:
      t.waiters[k] = newSeq[ptr Future[V]]()
    var fut = newFuture[V]("AsyncTable.pop")
    t.waiters[k].add fut.addr
    result = await fut

proc put*[K, V](t: AsyncTable[K, V], k: K, v: V) {.async.} =
  withLock(t.lock):
    if k in t.waiters:
      var ws: seq[ptr Future[V]]
      doassert t.waiters.pop(k, ws)
      while ws.len > 0:
        let w = ws.pop()
        if not w.isnil and not w[].isnil and not w[].finished:
          w[].complete(v)
    else:
      t.table[k] = v

proc requestTask(q: sink ptr Request) {.async.} =
  var trial = 0
  var
    sess: HttpSessionRef
    req: HttpClientRequestRef
    resp: HttpClientResponseRef
    cleanup: seq[Future[void]]
  while trial < q[].retries:
    try:
      trial.inc
      sess = new(HttpSessionRef,
                proxyTimeout = 10.seconds.div(3),
                headersTimeout = 10.seconds.div(2),
                connectTimeout = 10.seconds,
                proxy = if q[].proxied: selectProxy(trial) else: "",
                flags = if q[].proxied: proxiedFlags else: sessionFlags
      )
      req = new(HttpClientRequestRef,
                sess,
                sess.getAddress(q[].url).get,
                q[].meth,
                headers = q[].headers.toHeaderTuple,
                body = q[].body.tobytes,
        )
      resp = await req.fetch(followRedirects = q[].redir, raw = true)
      checkNil(resp):
        defer:
          cleanup.add resp.closeWait()
          resp = nil
        q.response.code = httpcore.HttpCode(resp.status)
        checkNil(resp.connection):
          q.response.body = bytesToString (await resp.getBodyBytes)
          q.response.headers = newHttpHeaders(cast[seq[(string, string)]](
              resp.headers.toList))
        break
    except:
      cdebug():
        logexc()
        debug "cronhttp: request failed"
    finally:
      if not req.isnil:
        cleanup.add req.closeWait()
      if not resp.isnil:
        cleanup.add resp.closeWait()
      if not sess.isnil:
        cleanup.add sess.closeWait()
  httpOut[q] = true
  await allFutures(cleanup)

O que eu faria diferente?

  • Provavelmente reescrever tudo em ferrugem, nim atualmente não lida bem com a segurança da memória, e a quantidade de tempo que tive que confiar no gdb para consertar travamentos foi demais, e nem consegui consertar todos eles. É um grande problema quando metade do ecossistema depende do GC e a outra metade do ORC (ou nem mesmo do orc e apenas do ARC). Misturar async e threads também é doloroso e stacktraces assíncronos são um pesadelo (embora eu não saiba se a ferrugem é melhor nesse aspecto).

  • Alvo um PWA desde o início. O projeto teve um início bastante problemático. No começo era para ser páginas estáticas servidas por um servidor web, então se tornou um servidor web propriamente dito. A interatividade veio como uma reflexão tardia, então se tornou apenas uma mistura de html renderizado mais js/css. Isso me deixou muito relaxado com a API, que saiu sem nenhuma estrutura (completamente inquieto ). Na reescrita, eu usaria uma estrutura de interface do usuário, ou pregar que tem suporte AMP completo, ou solidjs.

  • Adicione mais analisadores ad hoc para plataformas populares. A análise simples de artigos não funciona muito bem (ou não funciona) quando as plataformas mais populares hoje em dia oferecem muito pouco conteúdo e muitos vídeos e imagens, portanto, a extração deve ser mais direcionada para mídia avançada em vez de apenas texto, se não for considerados ao planejar a arquitetura do scraper, o conteúdo e as informações que o APP servirá serão desbalanceados.
