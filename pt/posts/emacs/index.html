Por que o emacs e minha lista de desejos pessoais do emacs.

O que é emacs?

UMA máquina virtual ou intérprete para o emacs linguagem de programação lisp ... mais ou menos. A linguagem de programação emacs lisp (elisp) é de propósito geral, mas tem suporte de primeira classe para o editor de texto real que a executa. Os tipos primitivos são focados na edição de texto, no entanto, você pode escrever o que quiser com desempenho decente (recentemente graças ao compilação jit ) porque é um descendente ceceio de MACLISP e um irmão de ceceio comum . No entanto emacs lisp não tem um padrão, e a especificação é equivalente à implementação mais popular, que é GNU Emacs.

Por que emacs?

Com a frase "Emacs é um sistema operacional", as pessoas querem dizer que você pode usá-lo para fazer qualquer coisa para a qual usaria um computador. Ele fica aquém de ser um sistema operacional literal, uma vez que o kernel (normalmente o Linux) ainda é o responsável pelo hardware.[1] No entanto, pode ser facilmente chamado de ambiente de área de trabalho, pois fornece um ambiente de trabalho. Ele oferece a capacidade de escrever código interativo rápido para fazer interface com qualquer aplicativo, pode ser tão sujo quanto scripts de shell ou bem planejado para fornecer APIs estáveis. Na verdade, muitas funcionalidades do emacs são fornecidas por pacotes.[2] Outros editores podem ser tão extensíveis quanto o emacs? Não por que? Porque outros editores usam um modelo de usuário diferente, em que o usuário deve ser restrito por ser considerado um "convidado" do ambiente de execução que está oferecendo um "serviço" ao usuário. Este não é o caso do emacs, onde o ambiente e o usuário tornam-se um e o mesmo. No emacs, você tem acesso a tudo e pode modificar quase todas as coisas pelos mesmos meios que essas coisas foram criadas, também conhecido como elisp.

Vale a pena?

Os benefícios se acumulam, é um investimento, e como qualquer investimento, você deve esperar ganhos proporcionais ao quanto você investe, portanto, você tem que gastar algum tempo aprendendo e praticando como usá-lo.

Minha lista de desejos GNU emacs

Emacs é um sistema operacional muito bom, mas um editor de texto ruim.

...Ou algo nesse sentido. A implementação do GNU Emacs, assim como o emacs, tem suas raízes nos anos oitenta, e muitas partes de seu núcleo mostram sua idade ...

  • A dicotomia entre a interface gráfica e a interface do terminal. Como o emacs é antigo, ele começou como um aplicativo para rodar dentro de um terminal, mas com o advento das interfaces gráficas nos anos 90, ele começou a coletar funcionalidades mais voltadas para baseado em janela interações e por volta do ano 2000, o suporte para o X11 protocolo foi introduzido. Com o passar dos anos, a utilidade da versão do terminal pode diminuir e, com isso, o interesse em mantê-la, porque ter o emacs se comportando de maneira diferente dependendo do gráfico / terminal é um fardo e um chip de complexidade. Eu gostaria de ver uma abstração da GUI do emacs sobre algo como o Sixelprotocolo, de modo que as interfaces gráficas e de terminal possam atingir um nível mais profundo de compartilhamento de código. Isso exigiria que o emacs invertesse a suposição de "primeiro terminal, segundo gráfico" (que, no momento da redação, acho que ainda se aplica).[3]

  • O código assíncrono em elisp não é fácil ou possível em alguns casos. Emacs suporta threads que produção isto é, pode pausar e devolver o controle ao agendador principal, que é uma implementação de multitarefa cooperativa . O emacs precisa de suporte moderno para primitivos assíncronos? O fato de o emacs ter sido restrito principalmente a um núcleo significa que o código é bastante otimizado, e o próprio emacs pode ser executado com bastante rapidez em dispositivos menores também. Eu acho que o emacs, sendo uma interface interativa, deve suportar apenas um interface gráfica assíncrona . Nesse caso, o escopo e o contexto do código assíncrono seriam restritos apenas à IU. Este é o design usado em outros editores baseados em elétrons, como VSCode , em que a IU é executada em seu próprio processo separado. Eu particularmente gostaria de experiência de digitação para nunca ser interrompido, e estou bem trocando realce de sintaxe atrasado, conclusão ou outras ferramentas auxiliares para soft realtime latência de entrada.

  • Já que no emacs você deve ter acesso a tudo , a maneira como o emacs fala o X11 é bastante ganancioso, visto que o emacs quase reimplementa um servidor X11 completo dentro de si. Isso causou, historicamente, alguns problemas[4] já que na maioria das vezes o emacs é usado como uma janela dentro de outro X11, e não um Próprio servidor X11 . O servidor de exibição, que lida com as sessões gráficas. Qual seria a aparência de um ambiente de exibição diferente? Eu imagino algo que começa de um nível inferior e se parece mais com um motor de jogo, talvez baseado em SDL . Além da saída de vídeo direta básica, que seria acessível a partir do elisp (compilado nativamente) ou de uma API C estável (ou ferrugem). O sistema de janelas emacs seria implementado no topo do backend "pixel cruncher" e não dependeria mais de um kit de ferramentas gráfico externo como lucid ou GTK e alcançar um nível mais alto de agnosticismo de plataforma. Isso também significaria um desempenho mais confiável entre os sistemas, porque quando a exibição é controlada por um kit de ferramentas gráfico externo, o desempenho pode se tornar imprevisível e, por exemplo, coisas como gerando um novo quadro acabam mais lentos no Windows do que no Linux; quando essas funcionalidades passam a ser usadas com frequência, como os candidatos de conclusão de tela, as discrepâncias tornam-se evidentes.[5]

  • Ter plataformas de exibição mais modernas permitiria não apenas melhor desempenho, mas acesso a ferramentas visuais mais avançadas, como transições adequadas, sombras, desfoque e um sistema melhor para camadas, já que o emacs atualmente as implementa com sobreposições que escalam mal. Em geral, seria bom ter todo o poder do CSS3 e além, pronto para usar e melhorar as interfaces emacs atualmente um tanto espartanas.[6]

  • Trabalhe em algumas especificações para elisp. Os mantenedores atuais do emacs se preocupam com a capacidade de executar o emacs em diferentes plataformas (arquiteturas de CPU). Se tanto do núcleo for reescrito no próprio elisp[7], a única coisa que resta nos internos torna-se o intérprete elisp. Com um acordo sobre alguma forma de especificação elisp, portar todo o emacs para um tempo de execução diferente se torna viável, uma vez que o problema é equivalente a "apenas" implementar um interpretador elisp.

  • Uma plataforma de financiamento emacs? Nisso reddit post é mencionado como é difícil "apenas financiar" pessoas para trabalhar nas coisas, e devem ser desenvolvedores interessados ​​em trabalhar em recursos específicos que precisam pedir financiamento. Isso, entretanto, não leva em consideração o fato de que muitas pessoas não são bem versadas nos componentes internos do emacs. Efetivamente, esperar que os desenvolvedores obtenham o conhecimento para implementar recursos específicos pode levar 20 anos (e contando); o que iria acelerar as coisas seria primeiro pagar as pessoas para aprender (e escrever documentação!) sobre os componentes internos do emacs de forma que mais pessoas possam se familiarizar com ele e mover lentamente o desenvolvimento do emacs do catedral e bazar modelo de desenvolvimento para um mais inclusivo.

  • Lista de desejos de pacotes:

    • Sentar na árvore todas as coisas! O realce da sintaxe do Emacs é baseado em expressões regulares, que não são lentas (na maioria das vezes), mas podem causar alguns declínios de desempenho que seria desejável evitar completamente. Mas o treesitter não apenas ajuda no realce da sintaxe, ele fornece uma representação hierárquica do conteúdo do buffer que beneficia outros pacotes também:

      • smartparens, paredit, lispyville, etc ...: Este é o pacote que fornece a API com mais recursos para aplicar manipulações de texto com base em objetos de texto, como slurping, mas eles são lentos (smartparens em particular). Rebasear a API no topo da treeitter parece atraente.

      • Polymode: é um modo secundário para misturar diferentes modos principais no mesmo buffer. Ele aplica o estreitamento da região por meio de expressões regulares, usando treesitter aqui ajudará a acelerar as coisas.

      • Orgmode: depois que uma gramática para o tree sitter foi escrita, o orgmode pode receber consideráveis ​​reduções de velocidade, uma vez que grande parte da lógica e da execução do modo org é baseada no conteúdo do buffer que deve ser regularmente analisado (propriedades, blocos de código, cabeçalhos, saídas)

    • Adicionar hipérbole to orgmode: este pacote implementa hyperlink globalmente, enquanto os hyperlinks orgmode funcionam apenas entre buffers orgmode. Também orgmode, que é baseado no modo de contorno, pode se beneficiar do modo kline fornecido pela hipérbole, que implementa nós não colidentes.

    • Circhat: um mashup de palavras entre circe.el e weechat.el . Circe implementa uma interface de usuário muito boa para buffers de bate-papo, enquanto weechat.el fornece suporte para interface com o protocolo de retransmissão do weechat , mas tem uma IU pobre e com bugs. Usar o weechat para lidar com as comunicações de bate-papo alivia o emacs de ter que lidar com todas as conexões de IRC e monitorar todos os buffers, o que pode diminuir bastante o emacs. Além disso, o weechat tem suporte a plug-ins para matriz, para que você também obtenha suporte direto de matriz dentro do emacs.

    • Tramp: tramp é o pacote emacs que lida com conexões remotas, mas é terrivelmente lento. Isso não é necessariamente falha de todos os tramps, porque outros pacotes nunca verificam se a sessão atual é local ou remota e aplicam coisas como operação pesada do sistema de arquivos em terminais remotos. Mas tramp também carece de suporte assíncrono, o que é uma situação muito triste para algo que principalmente envia comandos para processos externos que lidam com os protocolos solicitados (ssh, sftp, docker ...). A experiência do VSCode com a remoção de hosts é muito melhor, e o emacs precisa melhorar seu jogo :)
