Algumas notas sobre a prova de trabalho (derivada de hashcash)

um breve resumo das categorias para gerar provas de trabalho

A prova de trabalho historicamente começou com implementações de CPU, depois com GPU e, em seguida, com ASICS cada vez mais eficiente graças à redução e otimizações, a menos que o algoritmo seja projetado para ser um pouco resistente o que significa que ele pode tentar saturar algo que funcione como o gargalo para evitar o benefício de otimizações futuras.

Algumas idéias e aplicações de gargalos notáveis:

  • Memória rígida e limitada : dureza é mais fácil, mas o progresso da tecnologia torna uma escolha ruim a longo prazo, enquanto estar limitado pela latência significa ter um fisica limite às melhorias possíveis, mas é mais difícil de projetar e está ligada à complexidade do algoritmo. [Esquemas de memória rígida] como Yescrypt, Argon2, Lyra2 foram usados ​​para prova de trabalho, punhal-hashimoto e ethhash também são difíceis de memória. Não existem realmente aplicativos à solta para algoritmos de limite de memória, além de prova de armazenamento, exceto que, neste caso, a pesquisa é o alvo e não apenas uma propriedade útil de alghoritmo de hash, porque as provas, neste caso, são baseadas em protocolos de desafio / resposta (.. desde que você tem que provar que está armazenando algo e não apenas fingindo armazená-lo ..)

  • Limite de largura de banda: a movimentação de dados entre lugares é limitada à capacidade do canal usado. Implementar um algoritmo de limite de largura de banda significa transmitir dados e aplicar um filtro (simples!) Sobre eles para descobrir padrões úteis. A força do algoritmo depende da incapacidade de um invasor de prever os índices onde os dados desejados podem estar localizados. As únicas implementações que conheço são encontradas em flor de neve onde um progressivamente enorme banco de dados gerado com muita entropia é usado como base. A desvantagem de ser limitado pela largura de banda é que é amigável para escala , que é a capacidade de empilhar e empilhar hardware para atingir capacidades de nível industrial.

  • Limite de complexidade : esta é uma definição muito ondulada para dizer que é difícil otimizar em relação a algo que parece aleatória ... pelo menos na superfície. Se você tiver alguma lógica que produza uma saída desejada e estiver tentando otimizar essa lógica, só poderá fazer isso se puder prever o comportamento da lógica para, por exemplo, valores de cache ou criar pipelines rápidos para sub-rotinas conhecidas a recorrentes; isso significa que a força do algoritmo depende e sua capacidade de falso aleatoriedade. Como exemplo a ideia de mineração de contratos inteligentes , ou qualquer outro algoritmo que busca dados de um blockchain. [RandomX] como usado em monero usa como semente o hash de ~ 50 blocos atrás, um precursor de randomX pode ser encontrado em adaptativo cryptonight ou randomjs. A aleatoriedade é apenas uma parte da implementação, a outra é ajustar a distribuição de probabilidade para corresponder às capacidades (por exemplo, conjunto de instruções) da máquina alvo desejada, no caso de randomx é uma arquitetura de CPU comum, para progpow é uma GPU (nvidia?: P), no caso de contratos inteligentes, o espelhamento lógico de mineração não é procurado, uma vez que as otimizações do processo subjacente têm utilidade pingando.

  • Limite de execução : ou Tempo vinculado, geralmente significa exigir algumas informações para produzir algumas informações, é claro que a sequencialidade (como em repetir a mesma coisa várias vezes) é um primitivo de todas as coisas criptográficas e é usado na maioria (todas?) cifras, uma vez que é a maneira mais direta (e óbvia) de trocar computação por ofuscação. Ao utilizar este conceito em um nível inferior (para criar uma função de hashing não paralelizável), o algoritmo deve ser o mais simples possível, como um baixo operação matemática, de modo que a possibilidade de divisão de trabalho através de processos simultâneos é minimizado. [VDF] s conseguem isso, embora, mesmo sem os requisitos de configuração, tais esquemas sejam deterministao que significa que "o mais rápido sempre vence", é por isso que um VDF só pode ser considerado como POW se uma implementação for conhecida (comprovada?) por não ser improvável por mais do que o atraso Δ (delta) de comunicação do sistema que pretende use-o (por exemplo, uma rede p2p). Qualquer algoritmo que visa o limite de execução sem um alçapão significa um cálculo 1: 1 entre o provador e o verificador, mas não é determinístico, o que é um requisito (ouso dizer ...) para o POW. [Unprll] era uma moeda usando hashing sequencial, a verificação de hashes era parcial (1/8?), Para melhorar a proporção de verificação / provador acima de 1. Outra moeda que parece usar hashing sequencial é pascalcoin com uma mistura de dureza de memória.
